# 第三部分 消费与操作 Drupal 8

在第 2 部分中，我们讨论了任何解耦式 Drupal 架构中最重要的构建块，从公开 Drupal 内容的 Web 服务到可作为 API 消费者基础的参考应用程序。在此过程中，我们还审视了 Drupal 中 API 优先的发行版、身份验证方法，特别是促进 Drupal 8 解耦的核心模块和贡献模块。

在这些章节中，我们将从“是什么”转向“怎么做”，通过提供如何发出请求以满足各种需求的全面视角，弥合客户端应用程序与其所消费的 API 之间的差距。我们将研究如何通过检索和操作 Drupal 实体，直接应用 Drupal 及其贡献生态系统提供的 Web 服务。通过这种方法，我们将运用对 Drupal 中 Web 服务生态系统（核心 REST、JSON API、RELAXed Web Services 和 GraphQL）的理解，来演示它们之间的相似之处与不同之处。我们还将探索核心 REST 的内置功能——Views REST 导出，它允许快速构建只读 API。

与前几章类似，我们首先转向核心 REST，它引入了 Drupal 自己的 `X-CSRF-Token` 标头，并允许在请求期间自定义序列化格式，从而提供 JSON、XML 或其他响应。在此过程中，我们还将探索 Drupal 中独特的内容建模特性，以及如何利用这些特性（借助 Drupal 核心模块 Views）设计丰富的内容 API。

尽管核心 REST 有时可能为构建消费者应用程序的人提供不太理想的体验，但它构成了 Drupal 中许多其他 Web 服务解决方案的基础，并允许架构师开箱即用地提供一个强大的支持 CRUD 操作的 API，无需额外编写代码。Views REST 导出也是如此，这意味着 Drupal 站点本身就可以提供一个功能完备的 RESTful API，而无需安装其他模块。

然后，我们将注意力转向为 Drupal 8 提供 Web 服务的贡献模块，即 JSON API、GraphQL 和 RELAXed Web Services。这些模块中的每一个都为消费者应用程序的开发者提供了一套不同的特性，因此它们的数据访问机制也各不相同。特别是，由于 GraphQL 不能被视为典型的 RESTful Web 服务，并且需要学习其规范，因此我们以不同的方式对待这些 Web 服务模块。

除了被其他开源社区广泛采用以及提供检索和操作内容的强大手段之外，JSON API 在检索关系和对 Drupal 的读取查询执行复杂过滤方面也提供了明显更好的体验。这意味着我们不仅可以获取内容实体，还可以获取其作者的用户信息，而这在核心 REST 中需要两个连续的请求。尽管 JSON API 以查询字符串为重点的方法可能会很快变得错综复杂，但在所有可用的 Web 服务解决方案中，它在 Drupal 社区中拥有最高的采用率以及最广泛的周边工具生态系统，我们将在第 23 章中看到这一点。

有时，将 RESTful API 与我们 Drupal 架构中用于其他目的的其他工具和特性配对也是有用的。RELAXed Web Services 就是这种情况，它是 Drupal 对 CouchDB 规范的实现，也是用于内容暂存和同步的 Drupal Deploy 生态系统的一部分。与提供离线功能的客户端技术相结合，RELAXed Web Services 可以成为需要复杂编辑工作流的内容生态系统的强大解决方案。尽管如此，RELAXed Web Services 也可以单独使用。

最后，我们将审视在解耦式 Drupal 的 Web 服务领域中一个更有趣的角色：GraphQL。尽管 Drupal 对 GraphQL 的实现缺乏开箱即用的写入查询功能，但它对读取查询采用独特的、可定制响应的方式，可以改进其他 Web 服务要求我们发出的那些复杂查询。如今，GraphQL 模块有力地满足了典型读取查询中的一系列需求，例如排序、过滤和条件分组。

由于贡献模块发展迅速，我们将从 Drupal 8 中最稳定的 Web 服务解决方案开始介绍，即核心 REST。
