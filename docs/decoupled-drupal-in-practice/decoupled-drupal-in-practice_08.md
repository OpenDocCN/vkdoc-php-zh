# 5. 解耦 Drupal 的优势

在你自己的实现中解耦 Drupal 有许多优势，但对营销团队、开发者和企业而言，有几点尤其有价值：即内容联合（"一次编写，随处发布"）、架构上的关注点分离（结构化数据与展示相区分）以及流水线式开发。如果你拥有一个专注于 JavaScript 或除 Drupal 之外的其他前端技术的团队，那么将 Drupal 用作 Web 服务提供者尤其有益。

渐进式解耦 Drupal 有其自身的优势，本章将对此进行概述。这些优势包括流水线式开发，以及在没有多个故障点的情况下保持单体架构的能力，此外还能提供混合体验，即可以为了用户的利益提供不同层级的交互性。



### 内容联合

“一次编写，随处发布”正迅速成为全渠道格局中营销和出版领域的流行信条。越来越多的内容提供商和出版商寻求利用多样化的媒介，将内容投送到日益增长的体验场景中。在这些场景下，Drupal 是众多体验应用的中心枢纽，所有这些应用都从单一可信数据源接收内容，无论该枢纽是一个完整的 Drupal 站点（带有可选的渐进式解耦），还是一个完全解耦的 Drupal 内容仓库。

这里需要强调的是，在构建体验生态系统时，Drupal 并不需要完全解耦才能成功实现内容联合。许多单体且端到端的 Drupal 站点也会将数据暴露给其他应用消费。实际上，这正是 Drupal 项目负责人 Dries Buytaert 推荐架构师以解耦方式使用 Drupal 的方式：作为一个面向公众的 Drupal 站点，同时也作为消费者应用的中央数据源。这既保持了 Drupal 端到端的连续性，也保留了 Drupal 网站本身的 Drupal 前端优势。

### 关注点分离

长久以来，Web 开发一直面临着结构化数据及其表现层之间关注点分离的问题。结构化内容及其交付构成了 Drupal 的结构骨架，而模板及其逻辑则构成了 Drupal 内容的呈现层。即使在单体 Drupal 中，我们也很容易区分构成 Drupal 关键后端功能的层（例如，数据库本身，数据库抽象层）和其前端功能（例如，Twig 主题层）。

尽管在单体 Drupal 和渐进式解耦 Drupal 中，这种关注点分离是隐式的，用户基本不可见，但在完全解耦的 Drupal 中，它变成了由 Drupal 处理的结构化内容和由消费者应用处理的表现层之间的显式分离。因此，通过以 JSON 或 XML 格式将数据暴露给前端，呈现和美学成为消费者应用的领域，这些应用使用 Drupal 内容创建自己的体验，独立于默认的 Drupal 前端。

在完全解耦的上下文中，`RESTful Panels` 模块对这一范式提出了挑战，但将 Drupal 中的呈现逻辑与结构化内容一同注入，首先就挑战了完全解耦 Drupal 所宣扬的关注点分离。

## 以用户为中心的用户体验

改善用户体验的好处也许是解耦 Drupal 带来的最具争议性的优势之一，因为良好的用户体验源于有效的设计和开发，而非必然来自架构决策。尽管如此，解耦 Drupal 确实带来了改善用户体验的优势，因为通过采用解耦 Drupal 架构，能够打造出更好地满足用户需求的体验。

在这种情况下，用户体验问题与前端开发者体验密切相关（参见下一节）。例如，一个需要频繁重绘内容的交互式应用，在使用更擅长此项任务的 JavaScript 框架时，可能比在 Drupal 中更高效。然而，对于为移动用户构建的消费者应用，用户体验必须非常适合其任务，而 Drupal 的响应式设计可能不足以提供以用户为中心的体验。

在渐进式解耦 Drupal 中，使用 JavaScript 框架来增强 Drupal 提供的现有用户体验，无论是尽可能多地替换它，还是为单个页面的某个部分添加交互性。通过这种方式，可以以增量方式，选择性地将更以用户为中心的体验应用于现有的 Drupal 前端。

## 前端开发者体验

对于许多前端开发者，特别是那些精通 JavaScript 的开发者来说，Drupal 基于 Twig 驱动的前端可能很复杂，并且与他们过去使用的范式不同。一些团队主要由 JavaScript 开发者组成，对他们而言，现代前端开发需要大量使用 NPM 和其他 JavaScript 开发工具。此外，Twig 和 Drupal 的 Ajax 框架不适合业务相关人员可能要求的高度交互式应用。

解耦 Drupal（无论是完全解耦还是渐进式解耦）使开发团队能够将专业知识以差异化方式应用于所需功能。例如，使用开发团队非常熟悉的 JavaScript 框架可以加速渐进式解耦项目的开发。同样，采用带有 JavaScript 客户端应用的完全解耦 Drupal 实现，可以访问 JavaScript 框架的各种特性。

这一点尤其正确，因为 JavaScript 作为一种语言近年来取得了巨大进步，并且在交互式客户端环境中具有有利的定位。ES6（当前 JavaScript 版本，正获得更广泛的浏览器支持）中的新特性，如箭头函数、解构赋值、扩展运算符和类等，为开发者提供了比以往更愉悦的体验。

### 流水线式开发

在流水线式开发中，拥有不同技能组合的团队可以并行处理实施中的不同组件，而不会相互阻碍工作或损害项目完整性。解耦 Drupal 也为流水线式开发流程打开了大门，在这种流程中，前端团队可以针对一个仅用于测试但尚未实际完成的模拟 Web 服务 API 来构建应用，而后端团队则可以构建暴露该 API 及其背后底层流程的后端。

完全解耦和渐进式解耦的 Drupal 架构都允许团队以各自的速度产出成果。例如，虽然一个 Ember 开发者可能精通 Handlebars，但他或她可能不理解在 Drupal 中提供 Web 服务 API 的系统，这时 Drupal 开发者就可以发挥作用。通过这种方式，前端开发者不再受 Drupal 主题层复杂性的束缚，可以控制标记和渲染，而 Drupal 开发者则可以专注于他们在后端的专业知识，并构建强大的 RESTful API。

### 结论

在本章中，我们探讨了解耦 Drupal（无论是完全解耦还是渐进式解耦）所带来的部分回报。这些包括内容联合（全渠道格局有助于实现“一次编写，随处发布”的梦想）和关注点分离（结构化内容与其呈现方式相区分）。

此外，开发者和用户都能从根据其需求精调的体验中受益，无论是更高的交互性、特定的设备，还是一组技术。最后，对于完全解耦和渐进式解耦方法，也许最相关的是流水线式开发的前景，即不同专长的开发者可以并行处理不同的组件。
