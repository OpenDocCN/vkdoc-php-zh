# 第 1 章 引言

“云”是新兴的技术热词。似乎每个人都在谈论“迁移到云端”，但很少有人真正理解这意味着什么，更不用说如何利用它了。每一次技术变革，总会有人以为新技术能解决他们所有的问题，甚至都不先盘点一下这些问题到底是什么。许多技术都做出了承诺，其中一些甚至确实能兑现，但要让承诺成真，你必须以正确的方式使用这项技术。

本书主要面向那些希望开始将应用迁移到云端、并想了解如何入门以及有哪些可选方案的开发者。其次，本书也适用于那些希望熟悉云计算技术和理念的经理人，以便他们更好地理解可用选项，以及不同的开发决策会如何影响自身。本书以 PHP 为重点，但即使 PHP 并非您计划使用的部署语言，本书仍将为您提供在任何语言或平台下架构云应用所需了解的知识。

本书将讨论几种不同的云选项，但重点将放在基于 Linode 及类似云平台进行开发上，原因将在第 2 章中说明。

请注意，本书会对特定的基础设施供应商进行多次比较。我并不代表任何特定公司，也不保证这些比较是万无一失或永久有效的。但这些比较基于我的经验以及写作时的知识，我已尽最大努力提供真实可靠的信息。尽管如此，虽然本书讨论的通用考量不太可能改变，但具体的供应商、产品及其表现*确实*会随时间推移而变化。

此外，虽然设置、编程和配置任务的基本框架在很长一段时间内可能保持不变，但随着产品和平台的变化，具体步骤和截图可能与本书存在差异。不过，本书提供的步骤仍可作为可靠指南，帮助您了解不同类型云供应商能提供哪些服务。

## 1.1 预备知识

本书几乎没有预备知识要求。即使你对我们使用的特定工具不熟悉，本书也提供了足够详细的逐步指导，你应该能够相当轻松地跟上。

由于这是一本关于 Web 应用的书，它假定你熟悉 HTML、CSS 的基本知识，以及互联网工作原理的基础（例如域名、IP 地址等）。如果你对这些内容不熟悉，建议你先放下这本书，去阅读我早先的著作《编程新手从这里开始》。《编程新手从这里开始》并未涵盖所有你需要了解的主题，但如果你理解了其中的概念，就应该能够完成本书中的大部分示例。

本书中的代码基于 PHP，但代码足够简单，无论你熟悉哪种语言，都应该能跟上。选择 PHP 是因为用 PHP 编写简短易懂的 Web 应用很容易。例如，在工作中，我几乎总是用 Ruby on Rails 编程。然而，如果不熟悉整个 Ruby on Rails 系统，几乎不可能理解 Rails 应用。相比之下，PHP 的“魔法”成分比 Rails 少，但这让读者更清楚发生了什么。使用 PHP，尤其是在本书使用的这个层次上，我相信任何有其他编程语言经验的人都能理解其中的内容。

本书还假定你对数据库和 SQL 有基本的了解。这不是绝对要求，但数据库代码本身基本上不加解释。然而，即使是略懂 SQL 的人，也应该能一目了然地理解 SQL 代码。

最后，本书选用 Linux 作为操作系统。不过，即使你对 Linux 一无所知，本书也会提供所有需要输入的命令，因此你实际上无需了解 Linux 就能使用本书。附录 A 提供了常用 Linux 命令列表，供你想了解更多时参考。

选择 Linux 的原因是在云环境中安装和使用它非常容易。虽然也可以在云环境中使用 Windows，但 Linux 从一开始就是为此类应用而构建的。命令行虽然看似神秘，但实际上能让服务器管理快速且无痛。

此外，由于 Linux、PHP 和 PostgreSQL（我们的数据库系统）都是自由软件，因此使用它们无需考虑许可问题。你只需安装所需内容，然后继续生活。你无需担心何时该向谁付费，无需担心你的使用是否符合产品许可，也无需担心有人审计你的业务以验证合规性。使用自由软件，软件就只是一种工具——你可以安装它，然后忘掉它。

原则上我并不反对专有软件，但由于它会引入大量额外的管理问题，我尽量避开它。我不反对人们因工作获得报酬，但我确实在原则上反对无休止、不必要的麻烦——当你过度依赖许可糟糕的专有软件时，就会陷入这种困境。

本书专注于 CentOS 7 Linux 发行版，因为 CentOS 通常是一个相当稳定且健壮的 Linux 发行版，受到大多数供应商的支持，并且许多其他 Linux 发行版都以 CentOS 为起点。

## 1.2 排版约定

本书使用了一些简单的排版约定，你需要注意。当涉及代码、输入的命令、URL 或任何其他需要输入的文字时，本书会使用类似这样的字体：`type me here`。与命令名称相同的程序名（即你输入以使用的名称）也以这种方式书写，用户名和文件名也是如此。

然而，许多需要输入的内容都需要用你自己的值来替换，例如你部署的机器的 IP 地址。这些需要替换的内容以全大写单词表示。例如，要访问你自己的网站，你需要在浏览器中输入 `http://YOUR.DOMAIN.HERE/`，其中 `YOUR.DOMAIN.HERE` 指的是你自己网站的域名。这些内容的含义以及应替换成什么，在正文中都有描述。

请注意，PHP 实际上有一些默认的全大写变量名，我们将会用到。因此，`$_GET`、`$_POST` 和 `$_FILES` 都是 PHP 中真实的变量名，不应被替换。

本书介绍了如何使用各种第三方服务。在描述各种网站的用户界面元素（如菜单、按钮和字段名）时，这些项目通常会被放在双引号中。例如，要从大多数程序中打印文件，你可以进入“文件”（File）然后点击“打印”（Print）。

## 1.3 输入或下载代码

本书围绕一个简单的留言簿应用展开，使用不同的应用架构来构建和重构它。该应用的所有代码都包含在本书中。说实话，我认为亲自输入本书中的所有代码对你很有好处。代码量不大，而且这有助于你思考正在构建的内容。然而，我理解当你只是想运行示例时，这可能会变得乏味，而拥有完整、可运行的代码对于解决问题非常有帮助。

因此，我们在本书中涵盖的每个应用变体的所有代码都可以从 `www.github.com` 下载。仓库中的每个分支代表书中对应用的一次不同修改。使用该仓库，你只需要根据书中提到的环境特定更改进行调整，例如设置数据库服务器的 IP 地址。这些信息也包含在每个分支的 `README` 文件中。

你可以通过以下地址下载代码：

`https://github.com/johnnyb/cloud-example-application`

如果你选择亲自输入代码，请密切关注程序中空格的使用位置。代码完全按照应输入的方式编写，除非文中另有说明，否则换行也应保持原样。在错误的位置放置空格和换行，或者在需要的地方遗漏它们，是搞乱代码的必然方式。
