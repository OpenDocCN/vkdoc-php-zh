# 第 2 节 CRUD API 端点与面向操作的端点

## 基于 CRUD 的 API 端点

开发者通常使用 REST 方法构建 API，即专注于资源。为了公开客户端可以对资源执行的操作，我们通常遵循 CRUD 首字母缩写，即**CREATE**（创建）、**READ**（读取）、**UPDATE**（更新）和**DELETE**（删除）。每个词代表一种操作，这意味着对于每个资源，我们可以：

-   创建一个新元素

-   读取资源列表，列出当前元素

-   更新一个现有元素

-   删除一个现有元素

当我们提到资源时，可以将其视为元素的集合（例如，图书资源）。

在定义资源 API 调用时，我们通常为要公开的每个 CRUD 操作设置一个 HTTP 端点。让我们看一个图书资源的例子：

-   **GET** `/api/v1/book`

-   **GET** `/api/v1/book/{id}`

-   **POST** `/api/v1/book`

-   **PATCH** `/api/v1/book/{id}`

-   **DELETE** `/api/v1/book/{id}`

如上所示：

-   **GET**端点执行**READ**操作。

-   **POST**端点执行**CREATE**操作。

-   **PATCH**端点执行**UPDATE**操作。

-   **DELETE**端点执行**DELETE**操作。

这种方法涵盖了基本操作。当我们想要公开更复杂的操作时会发生什么？让我们在下一节中探讨。

## 面向操作的端点

想象一个电子商务 API，它有一个用于订单的端点（`api/v1/orders/{orderId}`），我们希望在此处批准客户下的订单。通过一个简单的 PATCH 请求，你可以发送如下的负载：

```
PATCH api/v1/orders/{orderId}
{
"status": "approved"
}
```

这种方法可行，但它没有明确传达更新背后的意图。服务器需要理解`approved`指的是批准订单，而不仅仅是更改一个通用的`status`字段。批准订单可能涉及比仅更新`status`字段值更多的任务。

让我们考虑将要执行的操作本身视为一个资源。为此，我们将创建一个订单操作资源，该资源会接收要执行的操作以及执行此操作所需的数据：

```
POST /api/v1/orders/operation
{
"operation" : "ApproveOrder",
"data" : {
"id" : 1559877444
}
}
```

这是一种更清晰的方法，因为负载中的操作名称明确传达了请求的目的。它帮助我们更进一步，通过操作端点指定业务操作。然而，由于我们只有一个端点，这种方法可能会使 API 文档的生成变得复杂。让我们考虑以下替代方案：

```
POST /api/v1/operation/approve-order
{
"id" : 1559877444
}
PATCH /api/v1/orders/:id/approve-order
```

上述替代方案将有助于创建文档，同时，我们的 API 将继续聚焦于操作。

在第一章的最后一部分，我们将看到如何以其他方式定义我们的端点，同时保持 API 面向操作。无论我们如何定义端点，发现、检索和执行操作的方式都是相同的。

这种方法需要发现要执行的操作，获取相关的处理器，并使用接收到的数据执行该操作。在本书的课程中，我们将学习如何使用 PHP 和 Symfony 框架对其进行编码。
