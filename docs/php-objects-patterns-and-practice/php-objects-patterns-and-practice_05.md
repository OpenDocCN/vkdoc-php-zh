# 1. PHP：设计与管理

电子补充材料 本章在线版本（doi:[10.1007/978-1-4842-1996-6_1](http://dx.doi.org/10.1007/978-1-4842-1996-6_1)）包含补充材料，仅供授权用户使用。

2004 年 7 月，PHP 5.0 发布。该版本引入了一系列重大改进，其中首要的便是对面向对象编程的显著增强支持。这激发了 PHP 社区对对象与设计的浓厚兴趣。事实上，这一过程早在版本 4 首次让面向对象编程在 PHP 中成为现实时便已开始。

本章将探讨使用对象编程所能解决的一些需求，并简要总结模式及相关实践演变过程中的一些方面。

同时，我还会概述本书涵盖的主题。

我将讨论以下内容：

- 灾难的演变：一个项目如何走向失败。
- 设计与 PHP：面向对象设计技术如何在 PHP 社区扎根。
- 本书的核心：对象、模式、实践。

### 问题所在

问题在于 PHP 实在太过易用。它会诱使你尝试自己的想法，并用良好的结果让你沾沾自喜。你将大量代码直接写入网页，因为 PHP 正是为此而设计的。你把实用函数（如数据库访问代码）添加到可从各页面引用的文件中，不知不觉间，一个可运行的 Web 应用便诞生了。

然而，你正一步步走向毁灭。当然，你并未察觉，因为你的网站看起来棒极了。它性能优越，客户满意，用户也乐于消费。

当你回头修改代码以开启新阶段时，麻烦便接踵而至。如今你拥有更大的团队、更多的用户和更充裕的预算，但毫无预兆地，一切开始出问题——仿佛你的项目中了毒。

新程序员艰难地理解着那些对你而言已是本能的代码，尽管这些代码可能因曲折复杂而略显古怪。她达到团队完全战斗力所需的时间比你预想的更长。

一次预计一日的简单修改，最终却花了三天，因为你发现必须更新 20 多个网页。

某位开发者将他自己的文件版本覆盖了你早些时候对该代码所做的重要修改。这个失误直到三天后才被发现，而此时你已修改了本地副本。又花了一天时间收拾残局，同时导致另一位处理该文件的开发者被迫停滞。

由于应用广受欢迎，你需要将代码迁移至新服务器。项目必须手动安装，却发现文件路径、数据库名称和密码被硬编码在许多源文件中。迁移期间你不得不暂停工作，因为不想覆盖迁移所需的配置更改。预计两小时的工作最终变成了八小时，因为有人用了 Apache 模块 `ModRewrite` 做了些“巧妙”处理，而应用现在必须依赖该模块才能正常运行。

第二阶段终于上线。一切顺利了一天半。就在你准备离开办公室时，第一个错误报告出现了。几分钟后客户打来电话投诉。她的报告与第一个类似，但进一步检查发现，这是另一个导致相似行为的错误。你回忆起阶段初那个简单的修改，却导致项目后续部分必须进行大量改动。

你意识到并非所有必要的修改都已到位——要么是起初被遗漏了，要么是相关文件在合并冲突中被覆盖了。你匆忙修改以修复错误。匆忙之中你来不及测试这些修改，但不过是复制粘贴的简单操作，能出什么问题呢？

次日清晨你到办公室时，发现购物车模块整夜宕机。你昨晚最后时刻的修改遗漏了一个前导引号，导致代码无法运行。当然，当你熟睡时，其他时区的潜在客户正精神抖擞地准备在你的商店消费。你修复了问题，安抚了客户，然后召集团队开始新一天的“救火”。

这个关于编程者的日常故事或许有些夸张，但我曾亲眼目睹这些事反复发生。许多 PHP 项目从小规模起步，最终演变成难以控制的庞然大物。

由于表现层也包含应用逻辑，重复代码很早就开始蔓延——数据库查询、身份验证检查、表单处理等等，不断从一页复制到另一页。每当某段代码需要修改时，所有存在该代码的地方都必须同步更新，否则必然导致错误。

缺乏文档使代码难以阅读，而缺乏测试则让隐蔽的错误直到部署后才被发现。客户业务性质的变化常常导致代码偏离最初的设计目标，最终执行起根本不适合的任务。由于这类代码通常已演变成一团交织混乱的整体，即使并非不可能，也很难抽离并重写部分代码来适应新目标。

现在，如果你是一名自由 PHP 顾问，这些都不是坏消息。评估和修复这样的系统足以支撑六个月的昂贵意式浓缩咖啡和 DVD 套装。但更严重的是，这类问题往往决定着一个企业的成败。



## PHP 与其他语言

PHP 的惊人流行意味着其边界很早就受到了严峻的考验。正如你将在下一章中看到的，PHP 最初是一组用于管理个人主页的宏。随着 PHP 3 以及更大程度上 PHP 4 的出现，该语言迅速成为大型企业网站背后的成功驱动力。然而，在许多方面，PHP 诞生之初遗留的影响仍然延续到了脚本设计和项目管理中。在某些领域，PHP 仍然有着不公正的名声，被视作一种业余爱好者的语言，最适合用于展示任务。

大约在同一时期（世纪之交），其他编程社区中新的理念开始流行。对面向对象设计的兴趣激发了 Java 社区。既然 Java 是一种面向对象语言，你可能会认为这有些多余。当然，Java 提供了一种更易于顺应而非抗拒的编程粒度，但使用类和对象本身并不能决定特定的设计方法。

设计模式作为一种描述问题及其解决方案本质的方式，其概念最早在 20 世纪 70 年代被讨论。也许恰如其分的是，这个想法起源于建筑领域，而非计算机科学。到了 20 世纪 90 年代初，面向对象程序员开始使用相同的技术来命名和描述软件设计问题。由埃里希·伽玛、理查德·赫尔姆、拉尔夫·约翰逊和约翰·弗利西德斯（此后在本书中将以其亲切的昵称“四人组”指代）合著的关于设计模式的开创性著作《设计模式：可复用面向对象软件的基础》（Addison-Wesley Professional，1995 年）至今仍是不可或缺的。它所包含的模式是任何初入该领域者的必经第一步，这也是本书中大部分模式都源于此书的原因。

Java 语言本身在其 API 中部署了许多核心模式，但直到 20 世纪 90 年代末，设计模式才渗透到整个编程社区的认知中。模式迅速充斥了商业街书店的计算机区域，电子邮件列表和论坛上也爆发了第一场论战。

无论你认为模式是一种传递工艺知识的强大方式，还是很大程度上是空谈（考虑到本书的标题，你应该能猜到我对这个问题的立场），都很难否认它们所鼓励的对软件设计的重视本身是有益的。

相关主题也日益突出。其中之一是由肯特·贝克倡导的极限编程（XP）。XP 是一种项目方法，鼓励灵活、面向设计、高度专注的规划与执行。

XP 原则中突出的一点是坚持认为测试对项目的成功至关重要。测试应自动化、频繁运行，并且最好在目标代码编写之前设计好。

XP 还规定项目应分解为小（非常小）的迭代。代码和需求都应时刻接受审查。架构和设计应是一个共享且持续关注的问题，从而促成代码的频繁修订。

如果说 XP 是设计运动中的激进派，那么温和派则由我读过的最好的编程书籍之一很好地代表：《程序员修炼之道：从小工到专家》，作者安德鲁·亨特和大卫·托马斯（Addison-Wesley Professional，1999 年）。

XP 曾被一些人认为有点狂热崇拜的意味，但它源于最高水平的二十多年面向对象实践，其原则被广泛借鉴。特别是代码修订，即所谓的重构，被作为模式的有力辅助手段而采用。重构自 20 世纪 80 年代以来一直在演变，但在马丁·福勒于 1999 年出版的《重构：改善既有代码的设计》（Addison-Wesley Professional）一书中被系统化地编目，并定义了该领域。

随着 XP 和模式的崛起，测试也成为一个热门话题。强大的`JUnit`测试平台的发布进一步强调了自动化测试的重要性，它成为了 Java 程序员武器库中的关键武器。由肯特·贝克和埃里希·伽玛撰写的一篇里程碑式的文章《被测试感染：程序员热爱编写测试》（[`http://junit.sourceforge.net/doc/testinfected/testing.htm`](http://junit.sourceforge.net/doc/testinfected/testing.htm)）精彩地介绍了这一主题，并且至今仍具有巨大影响力。

大约在这个时候，PHP 4 发布了，它带来了效率的提升，并且至关重要的是，增强了对对象的支持。这些增强使得完全面向对象的项目成为可能。程序员们欣然接受了这一特性，这多少有些出乎 Zend 创始人 Zeev Suraski 和 Andi Gutmans 的意料，他们当时已加入 Rasmus Lerdorf 共同管理 PHP 开发。正如你将在下一章看到的，PHP 的对象支持远非完美。但通过严格自律和谨慎使用语法，人们确实可以开始同时用对象和 PHP 进行思考。

尽管如此，正如本章开头所描述的那样的设计灾难仍然很常见。设计文化还有一段距离，在关于 PHP 的书籍中几乎不存在。然而，在线上，兴趣是显而易见的。Leon Atkinson 在 2001 年为 Zend 撰写了一篇关于 PHP 和模式的文章，Harry Fuecks 于 2002 年在[`www.phppatterns.com`](http://www.phppatterns.com)（现已关闭）创办了他的日志。基于模式的框架项目如`BinaryCloud`开始出现，同时自动测试和文档工具也相继涌现。

2003 年，第一个 PHP 5 测试版的发布确保了 PHP 作为一种面向对象编程语言的未来。Zend 2 引擎提供了大幅改进的对象支持。同样重要的是，它发出了一个信号：对象和面向对象设计现在是 PHP 项目的核心。

多年来，PHP 5 不断发展改进，集成了诸如命名空间和闭包等重要新特性。在此期间，它巩固了作为服务器端 Web 编程最佳选择的声誉。

2015 年 12 月发布的 PHP 7 代表了这一趋势的延续。它特别提供了对标量类型声明和返回类型声明的支持——这是许多开发者（以及本书的先前版本）多年来一直呼吁的两种特性。如果你不明白这意味着什么或为何重要，请继续阅读！

## 关于本书

本书并不试图在面向对象设计领域开创新局面；在这一点上，它是岌岌可危地站在巨人的肩膀上。相反，我将在 PHP 的背景下审视一些成熟的设计原则和一些关键模式（特别是《设计模式》——经典四人组书中记载的那些）。最后，我将超越代码的严格限制，着眼于有助于确保项目成功的工具和技术。除了本篇引言和一个简短的结论之外，本书分为三个主要部分：对象、模式和实践。



### 对象

在第二部分的开始，我将简要回顾一下 PHP 与对象的历史，描绘它们如何从 PHP 3 中的事后考虑，转变为 PHP 5 中的核心特性。

即使对对象知之甚少，你依然可以成为一名经验丰富且成功的 PHP 程序员。为此，我将从基本原理出发，解释对象、类和继承。即便在这个早期阶段，我也会探讨 PHP 5 和 PHP 7 引入的一些对象增强功能。

在奠定基础之后，我将更深入地探讨我们的主题，研究 PHP 更高级的面向对象特性。我还会用一章的篇幅来介绍 PHP 为了帮助你使用对象和类所提供的工具。

然而，只知道如何声明一个类并使用它来实例化一个对象是不够的。你首先必须为你的系统选择正确的参与者，并决定它们之间交互的最佳方式。与关于对象工具和语法的生硬事实相比，这些选择更难描述和学习。我将以对 PHP 面向对象设计的介绍来结束第二部分。

### 模式

模式描述了软件设计中的一个问题，并提供了解决方案的核心。这里的“解决方案”并不意味着你在参考手册（尽管参考手册作为程序员的资源非常出色）中能找到的那种可直接复制粘贴的代码。相反，设计模式描述了一种可以用来解决问题的方法。可能会给出一个示例实现，但它远不及它所旨在阐述的概念重要。

第三部分首先定义了设计模式并描述了它们的结构。我还将探讨它们受欢迎背后的一些原因。

模式往往倾向于促进并遵循某些核心的设计原则。理解这些原则有助于分析模式的动机，并且可以有效地应用于所有编程。我将讨论其中一些原则。我还会研究统一建模语言（UML），它是一种描述类及其交互的平台无关方式。

尽管本书并非模式目录，但我仍会研究一些最著名且最实用的模式。我将描述每种模式要解决的问题，分析其解决方案，并给出一个 PHP 中的实现示例。

### 实践

即使是一个设计得再优雅的架构，如果管理不当也会失败。在第四部分，我将探讨可用于帮助你创建一个确保项目成功的框架的工具。如果本书其余部分是关于设计和编程的实践，那么第四部分则关乎管理代码的实践。我所研究的工具可以为项目形成一个支撑结构，帮助跟踪出现的错误，促进程序员之间的协作，并提供易于安装和代码清晰性的便利。

我已经讨论过自动化测试的强大功能。我将以一个介绍性章节开启第四部分，概述该领域的问题和解决方案。

许多程序员都有一种冲动，即凡事都自己动手，这通常是一种错误。`Composer` 及其主要仓库 `Packagist` 提供了对数千个依赖管理包（可以轻松地集成到项目中）的访问。我将审视自行实现某个功能与部署一个 `Composer` 包之间的权衡。

既然谈到了 `Composer`，我将介绍一种安装机制，它使得部署一个包就像执行单个命令一样简单。

代码关乎协作。这一事实既可能带来收益，也可能变成一场彻头彻尾的噩梦。`Git` 是一个版本控制系统，它允许多个程序员在同一个代码库上协作，而不会相互覆盖彼此的工作。它让你能够在开发的任何阶段为你的项目拍摄快照，查看谁做了哪些更改，并将项目分割成可合并的分支。`Git` 终有一天会拯救你的项目。

当人和库进行协作时，他们常常会带来不同的约定和风格。虽然这很健康，但也可能破坏互操作性。像符合（conform）和遵从（comply）这样的词会让我不寒而栗，但不可否认的是，互联网的创造力正是建立在标准之上的。通过遵守某些约定，我们得以在一个难以想象的广阔沙盒中自由发挥。因此，在一个新章节中，我将探讨 PHP 标准、它们如何帮助我们，以及我们应该如何以及为何要，是的，遵从标准。

有两个事实似乎是不可避免的。首先，错误常常在同一段代码区域中重复出现，使得某些工作日像是一次“既视感”的练习。其次，改进常常会破坏与它们修复的同等或更多的东西。自动化测试可以同时解决这两个问题，为你的代码提供问题的早期预警系统。我将介绍 `PHPUnit`，它是所谓的 xUnit 测试平台的一个强大实现，该平台最初为 Smalltalk 设计，但现已移植到多种语言，尤其是 Java。我将特别关注 `PHPUnit` 的特性，并更广泛地讨论测试的益处以及一些代价。

应用程序是杂乱的。它们可能需要将文件安装在非标准位置，或者想要设置数据库，亦或是需要修补服务器配置。简而言之，应用程序在安装过程中需要做一些事情。`Phing` 是一个名为 `Ant` 的 Java 工具的忠实移植版。`Phing` 和 `Ant` 会解释一个构建文件，并按照你指示的任何方式处理你的源文件。这通常意味着将它们从源目录复制到系统周围的各种目标位置，但是，随着你的需求变得更加复杂，`Phing` 能够毫不费力地扩展以满足这些需求。

一些公司会强制执行开发平台——但在许多情况下，团队最终会运行各种不同的操作系统。承包商带着 PC 笔记本电脑来了（你好，Paul Tregoing，第五版技术编辑），一些团队成员喋喋不休地宣扬他们最喜欢的 Linux 发行版（那是我和我的 Fedora），而许多人则坚持要另一台看起来很酷的 PowerBook（在咖啡吧和会议室使用它根本不会让你看起来像是潮人 Borg 大军中的又一个节点）。所有这些系统都能以不同程度的易用性运行 LAMP 环境。然而，理想情况下，开发人员应该在与最终生产系统非常相似的环境中运行他们的代码。我将介绍 `Vagrant`，这是一个利用虚拟化的应用程序，以便团队成员可以保留他们特有的开发平台，但在类似生产环境的系统上运行项目代码。

测试和构建都很好，但你必须安装并运行你的测试，并且持续这样做才能获得收益。如果你不自动化你的构建和测试，很容易变得自满并让事情停滞不前。我将研究一些被归入“持续集成”这一类别的工具和技术，它们将帮助你做到这一点。


## 第五版新增内容

`PHP` 是一门不断演进的编程语言，因此它始终处于持续的审查与开发之中。本新版也经过了审查和全面更新，以反映语言的变化和新的可能性。我涵盖了匿名类等新特性，以及期待已久的标量参数类型提示和返回类型声明。示例中适当采用了 `PHP 7` 的特性，因此请注意，你需要使用 `PHP 7` 解释器运行代码——或者准备好做一些降级处理工作。

在之前的版本中，我包含了一章关于 `PEAR` 包仓库的内容。如今，`Composer` 和 `Packagist` 仓库显然已成为 `PHP` 开发的标准，因此我相应重写了这一章。

似乎本书每隔一个版本我就会切换一次版本控制内容。很高兴这次我坚持使用 `Git`。不过，由于像 `GitHub` 这样的 `Git` 仓库越来越多地被开发者使用，我花了一些更多时间来探讨它们。

## 总结

这是一本关于面向对象设计与编程的书。它同时也涉及从协作到部署管理 `PHP` 代码库的工具。

这两个主题从不同却又互补的角度解决了同一个问题。主要目标是构建既能实现目标，又适合协作开发的系统。

次要目标则在于软件系统的美学。作为程序员，我们构建具有形态和行为的机器。我们投入工作中的许多小时，乃至生命中的许多天，来将这些形态编写出来。我们希望构建的工具——无论是单个类与对象、软件组件，还是最终产品——都能构成一个优雅的整体。版本控制、测试、文档和构建的过程不仅支持这一目标：它本身就是我们想要达成的形态的一部分。正如我们追求简洁优雅的代码，我们也希望代码库对开发者和用户都设计良好。共享、阅读和部署项目的机制应与代码本身同样重要。
