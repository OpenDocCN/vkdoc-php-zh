### 关于作者

我从 2000 年底开始使用 PHP 编程。最初这并非我的选择，因为我当时偏爱 Perl 和 Delphi（也称为 Object Pascal）这两种语言。1999 年我从利兹大学获得计算机科学学位后，成为一名自由网页开发者，情况开始发生变化。仅仅几个月后，我就有机会从一位朋友手中接手一份（相对而言）规模庞大的政府网站合同，这位朋友正离开自由职业界，去拥抱更安稳、更理性的全职工作。唯一的障碍是，已有数千行代码写好了，而且是用一种名为“PHP”的相对较新的语言编写的。哦，另一个障碍是，我只有大约一周的时间来学习它才能接手网站。于是，按照当时的方式，我跑到当地的 Waterstones 书店（对于年轻读者来说，那是个买书的地方。纸质书。而且我们得亲自出门去买。或者在线下单，等上几天才能收到。我提过它们是纸做的吗？）。我手里拿着纸质版的《傻瓜通用 PHP 与 MySQL 入门完全指南合集》（书名我可能没记全），坐下来，品着一品脱的啤酒（记住，我当时在约克郡），开始阅读。几天后，我就能像专业人士一样写代码了（好吧，代码能跑通），12 年后的今天，我从未回头。在经历了多样的自由职业生涯，并创办了几家（IT 相关及不相关的）初创公司（成功程度不一，通常很惨淡）之后，我现在是牛津大学的一名全职程序员。我的日常工作涉及中等规模的数据采集与管理、统计分析，并为研究人员和公众提供用户界面。我的大部分开发工作都是用 PHP 完成的，要么开发新项目，要么整合他人的软件、系统和数据库。

在我的整个职业生涯中，我一直使用 PHP 进行网页开发，但对于桌面 GUI 工作，我最初使用 Delphi（然后是 Free-Pascal/Lazarus），并用 bash shell 脚本补充处理其他任务。这主要是因为我在大学时学了这些语言。然而，随着 PHP 的成熟，我越来越多地在网页之外使用它，现在我几乎不再用其他语言来处理遇到的任何编程或脚本任务。由于在大学期间以及一些自由职业工作中，我不得已接触过其他语言，如 C++、Javascript、Fortran、Lisp（可能还有我大脑故意不想记住的其他语言），我可以诚实地说，PHP 现在是我的首选语言，而非不得不用。

当我不被电脑束缚时，我很想说我有许多丰富多彩的兴趣爱好。我以前确实有。我可以写一整本书（销量不会好）来讲述我的经历和见闻，并且我愿意相信这些经历让我成为一个全面发展的人。但如今，我没有那么多爱好了。这很大程度上是因为我三个可爱的小女儿艾莉、伊兹和英迪，我美丽的妻子帕尔芙，以及我更加美丽的猫咪米娅对我的需求。而且，我不希望有任何改变。至少，我是这么告诉自己的……

### 1.1 “用 PHP？我们又不是在建网站，你知道吧！”

无论是它目前递归式的名称（“PHP：PHP 超文本预处理器”），还是其创造者拉斯马斯·勒德尔夫最初赋予它的名字（“PHP：个人主页”），都强化了人们普遍认为 PHP 是一种 Web 脚本语言的看法。这种看法在 1995 年 PHP 首次被创建时以及之后的多年里，确实是正确的。在 Web 领域，PHP 表现出色：它易于使用、开发迅速、部署广泛、与 Web 技术栈紧密结合（它通常是 LAMP、WAMP、MAMP 等中的 `P`），当然，还因为它免费且开源。

但许多人没有意识到（或者没注意到，或者选择不注意到）PHP 已经进化并成熟了。它现在非常接近于一种现代的、通用的编程语言。这种认知上的缺失部分要归咎于 PHP 自身，因为它在一些基础功能上花了很长时间才完善，例如面向对象的语言结构，甚至是在没有 Web 服务器参与的情况下也能运行的能力。然而，编程社区也没有起到帮助作用；许多程序员难以看到 PHP 将其快速动态开发模型从 Web 带入更广泛计算环境的潜力，许多人只是固守着“面向初学者的 Web 脚本”这种教条，而这实际上只在其早期阶段才是真实的。

特别是最近的几个版本，已经带来了主流的语言特性（例如，闭包、trait、更好的语言支持、命名空间和后期静态绑定等等）。性能不断提升，内存使用（旧版本的一个痛点）显著下降，PHP 现在已成为更精简的流行脚本语言之一。为了满足更高的性能需求，PHP 现在内置了 Op-code 缓存，这对于经常使用的脚本非常有用，而 Facebook（全球最大的 PHP 用户）是正在发布带有 JIT 编译器的替代解释器/虚拟机（VM）的几个团队之一，这使得性能比标准 Zend 引擎最高可提升 6 倍。当然，PHP 核心语言的开发仍在快速进行中，5.6 版本最近已发布，具有更好的性能、集成的调试器和众多语言增强功能，这表明有关 PHP 消亡的报道还为时过早！

如果你是一位希望在 Web 领域之外拓展工作的 PHP Web 开发者，那么现在正是利用你现有技能，尝试在 PHP 中进行桌面和系统编程的最佳时机。如果你是一位正在考虑学习哪种语言的新手程序员，你可以学习到的技能将能应用于比许多其他类似语言更广泛的场景。

但是，为什么，到底为什么要用 PHP 来做这些呢？为什么不写个 Shell 脚本？为什么不学习 C++ 或其他通常用于软件项目的语言？事实是，这些都是可行的选择，你的人生可能也会因此一帆风顺。但是，为什么要放弃运用你现有技能的机会呢？为什么不使用 PHP 集成的数据库访问、重用来自你 Web 项目的现有代码和数据、利用 PHP 易于使用的网络库和函数、享受灵活的数据和文本处理、以及根据需要将 Shell 命令和其他语言混入 PHP 中，从而两全其美呢？简而言之，为什么不用呢？

本书旨在向您，PHP 程序员，展示如何在 Web 之外使用 PHP。我们将探讨构建桌面应用、命令行工具、系统守护进程、网络服务器和其他本地应用程序。重要的是，我会向你展示如何在不离开 PHP 及其友好生态这个舒适圈的情况下完成这一切。

希望我已经让你相信，PHP 是一种跨平台、注重快速开发、功能多样的语言，非常适合许多不同类型的软件。但如果你仍然怀疑 PHP 交付真正非 Web 软件的能力，并且不想通读全书来了解其实现能力，那么请直接跳到书末的附录 D，那里有诸如 Web 服务器、数据库引擎、机器学习工具等完全用 PHP 编写的项目示例。如果这些都不能激励你用 PHP 创建自己的软件，那么可能没什么能了！

### 1.3 读者前提——或者说，这本书不是什么

要充分掌握本书内容，你需要具备基本的PHP编程经验（多数情况下是Web开发），拥有常规编程或IT背景，并且愿意学习、敢于走出舒适区。

本书并非PHP或通用编程的入门读物。虽然你不需要计算机学位或高级编程概念的知识，但本书的难度设定在普通PHP程序员（即对PHP的掌握不仅限于基础层面的人）的水平，并会在讲解过程中解释任何必要的概念。本书对于高级程序员同样有用，他们可以将其作为快速参考，探索Web之外PHP的应用。

书中提供了代码示例来阐述概念和函数用法，但通常不会提供完整的脚本或程序。本书作为参考和入门读物，旨在为你提供信息和灵感，帮助你开发自己的PHP解决方案。同样，在讨论外部工具和第三方库时，我通常会提供概述、基本介绍或示例代码，然后附上官方文档和在线教程的链接。像这样涵盖广泛领域的书籍，若对某个特定软件（尤其是那些已有详尽文档的）进行深入探讨和大篇幅介绍，意义不大。当然，对于官方或在线文档缺失的相关领域，我们会进行补充，并在必要时直接为你指明相关章节。

本书中的大部分示例均在Linux环境下（多数情况下为最新的Ubuntu桌面版/服务器版）运行和测试。PHP可在包括MS Windows和Apple OS X在内的多种操作系统上运行，代码通常能以完全相同的方式运行。当然，由于文件系统、操作系统、可用库等方面的差异，确实存在不同之处，但在本书中一一覆盖这些情况并不现实。此外，某些特性（例如依赖POSIX标准的特性）在某些系统（如Windows）上无法（轻易）使用。由于OS X源自符合POSIX标准的操作系统，你可能会发现与Windows相比，更多代码是兼容的，但实际体验可能仍有差异。关于操作系统特定问题的良好信息来源是官方PHP在线手册，特别是每页底部的用户评论。在可能的情况下，也会涉及其他操作系统的特定领域，例如我们会探讨如何访问Windows注册表。

## 2. 脱离Web——基础篇

本章概述了使用PHP摆脱Web束缚的基本步骤。我们将探讨涉及的技术步骤，以及编程实践和侧重点上的差异。

### 2.1 无Web服务器的PHP

大多数PHP程序员都严格在Web服务器环境下使用过PHP。在这种环境中，PHP作为CGI/FastCGI或服务器模块，由HTTP服务器（通常是Apache、IIS、Nginx或类似软件）调用和控制。HTTP服务器接收到对基于PHP的网页的请求后，会调用PHP进程执行该请求，这通常会将输出返回给HTTP服务器，再由其发送给最终用户。

人们曾多次尝试通过本地安装的Web服务器，用PHP创建本地应用程序（桌面应用和脚本系统），并取得了一些成功。然而，对这种基于Web的模型用于本地应用，其缺点众多：

-   就它们在这些场景中提供的价值而言，像Apache这样的Web服务器往往过于臃肿且消耗大量资源

-   除非经过妥善锁定，否则本地运行的Web服务器会为来自外界的恶意访问者提供一个访问你机器的入口点

-   HTTP是一种冗长的协议，并且（可以说）非常适合Web，但对于本地进程间通信来说，它往往过于复杂，并为你的应用程序增加了额外的资源和复杂性开销

-   你的应用程序界面通常在Web浏览器中运行，因此看起来一点也不像本地应用（并且会带来额外的支持/升级麻烦，除非你专门为你的应用安装一个浏览器）

自PHP 5.4版本起，它包含了一个内置的Web服务器，消除了上述部分问题。然而，该服务器是为本地测试那些设计用于在生产环境中、像Apache这样功能完备的HTTP服务器上部署的PHP脚本而设计的。对于某些本地应用，特别是当你希望运行某个已存在的PHP Web应用的私有实例时，这是一个值得探索的途径。然而，它对于生产级本地应用的稳定性、可靠性和适用性仍有待验证，并且它依然带有HTTP和Web浏览器的负担。

自4.3版本起，PHP在内部隐藏了一个解决所有这些问题的“金块”。`PHP CLI SAPI`（命令行接口服务器应用程序编程接口），其正式名称，本质上就是“独立PHP”的繁琐表达。它省去了对Web服务器的需求，并提供了一个独立的PHP二进制文件供你交互。例如，在shell提示符下输入：

```
~$ php /home/myfiles/myprogram.php
```

将简单地执行你的`myprogram.php`文件（它的编写方式与其他 PHP 脚本几乎完全相同），并将任何输出返回给终端（除非你另有指令），而不是发送给 Web 服务器（它根本不存在！）。

通常，直接使用 PHP CLI 调用的 PHP 脚本，其行为方式与通过 Web 服务器调用时相同，尽管存在一些差异。例如，你找不到任何`$_GET`或`$_POST`数组，并且 PHP 默认不会发送任何 HTTP 头，因为这些概念在 Web 之外意义不大。像`max_execution_time`这样的默认设置会被设置为适合本地使用的合理值（本例中为 0，因此你的脚本不会超时），输出缓冲被关闭，并且错误消息以纯文本而非 HTML 格式显示。

PHP CLI SAPI 将是我们在本书大多数示例中使用的，因为它的设计动机与本书完全一致：让 PHP 超越 Web。下一章将提供如何安装和使用 CLI SAPI 的完整细节。

### 2.2 PHP 版本——你用的是哪个？

自 4.3.0 版本起，PHP 就开始支持 CLI SAPI，因此本书中的许多示例可以在其后的任何 PHP 版本上运行。在撰写本文时，PHP 的当前版本是 5.6.5，本书中的代码已在此版本上测试过。如果你发现某个特定函数不存在或未能按预期工作，请查阅该函数的在线 PHP 手册。手册会指出哪些版本支持哪些函数，并描述新版本带来的任何破坏性变更。

如果你使用的是较旧版本，则有一些非常充分的理由进行升级。

-   在较新版本中，性能显著提升（相应地，资源占用也减少了）

-   虽然安全性对于非 Web 应用并非总是至关重要（有关注意事项，请参阅本章后面的安全讨论），但如果你正在处理来自外部的数据，或者你同时也在使用相同版本进行 Web 开发，那么新版本中的安全增强和安全相关错误修复是必不可少的

-   现代语言特性可用，这有助于提高你的编码效率，并帮助他人更认真地对待你的代码

与 PHP 的 Web 版本一样，如果你有需要，可以自己编译 CLI SAPI 的版本。如果你希望包含操作系统软件仓库未预打包的扩展、出于性能原因移除非必要代码，或使用 PHP 支持的任何其他编译时选项，那么学习“自制”版本可能会很有价值。关于编译和安装 PHP 及相关扩展的入门指南，请参见附录 A。

### 2.3 几个不使用 PHP 的充分理由

本书专注于用 PHP 完成所有事情，并且对于 90% 的任务，PHP 都能胜任。但在你承诺用 PHP 重写你的世界之前，有几个相关的问题你可能需要在开始之前仔细考虑。

-   **高性能要求**

如果你需要非常高的性能，特别是在有限的硬件上，PHP 可能不适合你。在 PHP 的较新版本中，性能取得了巨大的飞跃，并且在基准测试中经常击败 Python 和 Ruby on Rails 等语言和框架。然而，PHP 的核心是一种抽象化的脚本语言，对于某些任务，它永远无法达到像 C 这样的低级语言所能获得的性能，后者更接近硬件层面。不过，不要完全否定它处理所有高性能任务的可能性。如果你是从成本角度考虑性能，你可能会发现，通过 PHP 快速开发所节省的开发人员时间成本，超过了为获得所需性能而投入的额外硬件成本。请查看第 9 章，了解更多关于提升性能的信息，并了解如何从 PHP 与 C 等语言进行交互，以便你仍然可以获得 PHP 中 RAD（快速应用开发）的一些好处。

-   **不要（非得）重复发明轮子**

你*可以*用 PHP 编写一个 Web 服务器（示例请参见附录 D），但*现有的*服务器如 Apache 难道不能满足你的需求吗？对于许多或大多数基础设施方面的“痒点”，都存在（用多种不同语言编写的）软件可以满足你的“挠痒”需求。除非你真的需要其他功能，否则编写自己软件所花费的时间通常比学习和部署现有软件的时间要多。当然，对某些人来说，情况正好相反。“因为我能做到”也可以是一个有效的理由，并且编写软件总是一个宝贵的学习经历。

-   **保持源代码封闭**

PHP 是一种脚本语言，这意味着如果你编写软件是为了出售或部署到其他地方，你将向接收者公开你的源代码。许多人（包括本书作者）会认为“开源”，即使是商业上的开源，也是一件好事。但如果这不是你的风格，那么你需要付出更多努力来保护你的代码。网上有可用的源代码混淆器，它们使用各种技巧使你的代码难以（但并非不可能）阅读。此外，现在已有多种 PHP 编译器可用于生成二进制程序（尽管在语法和扩展覆盖方面存在限制）。但是，主要的 PHP 项目并未表达支持代码隐藏或编译的意图，因此此类方法的长期可行性尚不确定。从中长期来看，闭源软件的商业案例也并非板上钉钉。

### 2.4 安全性考量

每一位称职的程序员，至少都清楚构建网站和在线应用所涉及的安全问题。你会刻意将自己的代码暴露给公众、全世界，以及所有来访者（无论是善意还是恶意）。PHP 早期的一个缺陷是优先考虑易用性而非代码安全性（像`register_globals`这样的遗留功能引发的恐怖故事，只需谷歌搜索一下便能知晓）。随着更新、更安全的默认设置和`register_globals`等功能的弃用，PHP 在线上环境比以往任何时候都更安全。尽管大多数安全问题是由程序员而非语言本身引起的，但即便是最新的网络开发者，如今也似乎从一开始就具备了安全意识。

然而，当你踏入离线软件的世界时，情况就截然不同了。通常，我们认为软件仅供受信任的用户使用，仅部署在受信任的机器上，并且完全由良善用户控制。用户不会故意攻击软件或机器，他们使用的是自己的数据，功能也绝对不应受损。在开发桌面应用等软件时，安全性很少被考虑，更不用说在设计时将其作为特性规划出来，因为觉得“没必要”。

但现实并非如此。看看任何软件漏洞邮件列表，比如 BugTraq，你会发现大量桌面应用的漏洞，例如 Adobe Reader、Microsoft Word，甚至像 GIMP 这样的开源软件。事实上，即使在本地应用中，安全性也很重要，主要有两个原因。第一个是“典型用户”行为的长期问题。这对于从不犯错的聪明技术用户（比如亲爱的读者您）来说不是问题，但对于我们其他人使用的任何软件来说，一次不小心点击可疑邮件就可能导致灾难。第二个原因是，对于大多数机器而言，许多非网络应用并非真正的“离线”。即使桌面应用或系统守护进程本身不与网络、本地网络或其他外部服务交互，其连接的机器也必然插有网线，或激活了 WiFi/3G/4G 连接。你的软件不会运行在与外界隔绝的舒适小天地里。当然，除非是完美沙盒化的虚拟机。

软件安全性本身就可以写成另一本书（已有不少相关著作，请参考下方的扩展阅读部分），许多原则同样适用于系统软件和网络软件，因此你可以利用现有的基于 PHP 的网络安全实践知识来指导自己。在规划应用安全措施和监控时，应考虑以下面向用户和系统软件中常见的漏洞类型和攻击向量列表。

- **受损的数据文件**

  来自外部来源的受损文件（由用户故意或无意加载）。这些通常是数据文件，尤其危险的是被注册为特定文件类型默认查看器的软件，因为意外和恶意的文件激活更容易发生。

- **被恶意软件利用**

  恶意软件寻找可被利用的无辜软件来提升权限。像 PHP 代码这样的脚本化软件更容易被恶意软件重写或篡改，而且源码以非编译形式存在可能为恶意软件作者提供便利。

- **用户不当行为**

  合法用户的不当行为。例如，约翰·史密斯想在其与上司简·多伊共享的业务系统中查看她的文件或浏览历史。

- **权限提升**

  类似于合法用户的不当行为，这是合法软件的不当行为，要么意外、要么故意地试图获取并使用其不具备的访问权限。

- **依赖关系漏洞**

  PHP 的漏洞，以及其他依赖项和相关软件中的漏洞。你的软件当然可能完全不存在安全问题，但它必然依赖其他库、软件和 PHP 扩展，当然，别忘了还有 PHP 解释器本身。这些都可能包含安全漏洞和攻击向量。

以上列表列出了所有类型软件中常见的安全问题来源，而不仅仅是 PHP。通过在设计阶段规划这些风险，并在部署前进行测试，可以最大限度地降低风险。然后祈祷好运吧……

## 3. 理解 CLI SAPI 及其必要性

正如我们在第 2 章中提到的，你进行的大部分系统编程都将涉及使用 PHP CLI SAPI。因此，深入了解如何使用它、配置和运行它的选项，以及它与你所熟悉的基于 Web 的 SAPI 有何不同，就显得十分重要。幸运的是，差异很小，而且许多是直观的，不过仍然值得完整阅读本章，因为 CLI SAPI 是大多数代码运行的基础。

### 3.1 CLI SAPI 有何不同？

以下是 CLI SAPI 与标准 Web 实现之间的主要差异列表：

- 默认情况下，输出中不会写入任何 HTTP 头。这很好理解，因为在命令行中它们毫无意义，只会成为打印在真实输出上方多余的文本。如果你的输出之后要被传输到网络浏览器，你需要手动添加必要的头信息（例如通过使用 `header()` PHP 函数）。

- PHP 不会将工作目录更改为正在执行的 PHP 脚本所在目录。要手动执行此操作，请使用 `getcwd()` 和 `chdir()` 来获取和设置当前目录。否则，当前工作目录将是你调用脚本时所在的目录。例如，如果你当前在 `/home/rob` 目录下，并且键入了 `php /home/peter/some_script.php`，那么 PHP 中使用的工作目录将是 `/home/rob`，而不是 `/home/peter`。

- 任何错误或警告消息都以纯文本形式输出，而不是 HTML 格式化文本。如果你希望得到 HTML 格式的错误提示，例如当你正在生成静态 HTML 文件时，可以通过在你的脚本中使用 `ini_set('html_errors', 1);` 将 `html_errors` 运行时配置指令设置为 true 来覆盖此行为。

- PHP 会隐式地立即“刷新”所有输出，并且默认不进行缓冲。将输出直接发送到浏览器通常会损害在线性能，因此输出会被缓冲，并在缓冲区满时以最佳大小的块发送出去。离线时这不太可能成为问题，因此像 `print` 和 `echo` 这类结构产生的 HTML 块和输出会直接发送到 shell。当你等待后续输出时，无需使用 `flush()` 来清除缓冲区。如果你愿意，仍然可以使用 PHP 的输出缓冲函数来捕获和控制输出，更多信息请参考 PHP 手册中的“输出控制函数”部分。

- 没有设置执行时间限制。你的脚本将持续运行，直到它自行退出，即使它挂起，PHP 也不会终止它。如果你想为行为异常的脚本设置时间限制，可以在脚本中使用 `set_time_limit()` 函数来实现。

- `$argc` 和 `$argv` 这两个描述传递给脚本的命令行参数的变量会自动设置。这些变量将在本章后面部分详细讨论。

- PHP 定义了与同名的标准流相关的常量 `STDIN`、`STDOUT` 和 `STDERR`，并自动为它们打开 I/O 流。这些常量让你的应用程序能够即时访问“标准输入”（`STDIN`）、“标准输出”（`STDOUT`）和“标准错误”（`STDERR`）流。

### 3.2 CLI SAPI 安装

要使用 PHP CLI SAPI，你可能需要先安装它。附录 A 提供了安装（以及必要时编译）PHP 的详细信息。然而，你可能会发现，如果你已经安装了 PHP，它可能已经安装好了（通常位于 PHP 程序文件夹中一个名为 `sapi/cli` 的文件夹内），如果没有，它通常可以在现代操作系统软件仓库中找到（例如，在 Ubuntu 中有一个名为 `php5-cli` 的包，可以通过任何包管理器或通过命令 `sudo apt-get install php5-cli` 在命令行中安装）。如果它安装在命令行搜索路径中，在命令行中键入 `php -v` 将打印版本详细信息，确认它确实已安装。

### 3.3 PHP 命令行选项

PHP 二进程序会接受许多影响其操作方式的命令行选项/开关/参数。可以通过键入 `php -h` 查看完整列表。虽然有些仅适用于 CGI SAPI，但以下是在与 CLI SAPI 交互时使用的一些更有趣且常见的选项：

- `-f` *或* `--file`

此选项允许你指定要运行的脚本的文件名，并且是可选的。`-f` 选项的存在是为了与某些软件和脚本（例如自动化软件）兼容，这些软件可以编程方式调用命令行程序，但要求文件名参数以此种方式形成。它还允许在 Windows 上为 PHP 脚本轻松设置默认文件类型处理程序。上述命令两种形式在使用上的唯一真正区别在于解释传递给脚本的命令行参数时，我们将在下面的“你的脚本的命令行参数”部分中讨论这一点。大多数情况下，下面两行基本等效：

```
1 ~$ php -f myscript.php
2 ~$ php myscript.php
```

- `-a` *或* `--interactive`

交互式运行 PHP。这允许你逐行输入 PHP 代码，而不是执行已保存的 PHP 脚本。这种操作模式通常被称为“REPL”（Read-Eval-Print-Loop，读取-求值-打印-循环）。它不仅为测试和开发代码提供了交互式界面，还可以充当增强的 PHP 功能的 shell 或命令行，在下一章关于开发工具的内容中，我们将更深入地研究它和其他第三方 PHP REPL。

- `-c` *或* `--php-ini`

指定 PHP 将用于此应用程序的 PHP `ini` 文件。如果你在同一台机器上还使用 PHP 运行 Web 服务，这将特别有用，因为如果未指定，PHP 将查找各种默认位置以寻找 `php.ini`，并可能最终使用与你的 Web 服务相同的文件。通过为你的 CLI 应用程序专门提供一个 `php.ini` 文件，你可以“开放”各种更适合离线应用程序的限制。请注意，通过使用 CLI SAPI，无论你是否使用此选项指定自定义 `ini` 文件，PHP 都会自动覆盖多个 `php.ini` 设置。这些被覆盖的设置正是影响上述“CLI SAPI 有何不同？”中所述行为的那部分设置，虽然在这些情况下 `php.ini` 文件被忽略，但你可以在代码中直接使用 `ini_set()` 函数或类似方法恢复或更改这些设置。你也可以使用 `-d` 或 `--define` 选项来设置选项（例如 `php -d max_execution_time=2000 myscript.php`）。如果你要将软件部署到你无法控制的机器上（例如，如果你在销售软件供用户在其自己的机器上安装），那么使用这些机制之一来确保 PHP 将使用你期望的设置运行，而不是用户可能碰巧设置的设置，这样做是合理的。另请参阅下面的 `-n`。

- `-n` *或* `--no-php-ini`

这个选项告诉 PHP 根本不加载任何 `php.ini` 文件。如果你不希望随应用程序附带 `php.ini` 文件，而是希望直接在代码中使用 `ini_set()` 或类似方法设置所有设置，这会很有用。如果没有提供 `ini` 文件，PHP 将使用其默认设置，并且值得记住的是，这些默认设置可能会随 PHP 版本的不同而变化（并且过去确实如此）。你不应该依赖当前的默认设置适合你的应用程序。你可以使用 `php --ini` 来显示当未使用 `-n` 选项且未使用 `-c` 指定文件时，PHP 将查找 `ini` 文件的默认路径。

- `-e` *或* `--profile-info`

这将 PHP 置于“扩展信息模式”。EIM 会生成供性能分析器和调试器使用的额外信息。如果你没有使用需要此模式的性能分析器或调试器，则不应启用它，因为它会降低性能。有关性能分析器和调试器的更多信息，请参阅下一章。

- `-i` *或* `--info`


调用`phpinfo()`函数并打印其输出。该函数以纯文本格式（而非通常的 HTML 格式）输出关于 PHP 安装的*大量*信息（它会检测到我们正通过 CLI SAPI 调用它）。这对于排查安装本身的问题非常有用，同时也能提供版本信息、已安装扩展列表、相关文件路径等。与任何其他 shell 命令一样，其输出可以通过管道传递给其他命令，例如`grep`。因此，如果你想检查 PHP 二进制文件中是否启用了 IPv6，可以尝试：

```
1 ~$ php -i | grep -i "ipv6"
```

* `-l` *或* `--syntax-check`

解析文件，检查语法错误。这是一个基本的“lint”类型检查器，更高级的静态代码分析工具将在下一章讨论。请注意，此选项仅检查会导致 PHP 引擎失败的基本语法错误。更微妙的错误、程序逻辑问题以及运行时产生的错误不会被检测到。你的代码不会被执行，因此它有助于在运行可能更改数据并在失败时导致问题的代码之前，提前发现基本错误。即使在测试环境中运行此类代码，重置数据并为下一次测试做准备也需要时间，因此首先快速检查基本语法错误可以节省时间。某些集成开发环境和文本编辑器会在后台运行`php -l`，以便在你输入时高亮显示语法错误。例如，Github 的 Atom 编辑器中的 linter-php1 包就使用此方法对 PHP 代码进行实时 lint 检查。

* `-m` *或* `--modules`

列出所有已加载的 PHP 和 Zend 模块/扩展。这些是 PHP 编译时包含的模块，可能包括“core”、“mysql”、“PDO”、“json”等。这对于检查 PHP 安装是否具有应用程序所需的功能非常有用。你也可以在脚本中使用`extension_loaded()`函数或调用`phpinfo()`函数进行检查。`-m`提供的信息是上面`-i`标志所提供信息的一个子集，而`-i`（或`phpinfo()`函数）将返回有关模块配置、版本等的更多信息。

* `-r` *或* `--run`

运行作为参数提供的单行 PHP 代码，而不是从文件执行它。该行代码应包含在单引号内，因为像 bash 这样的 shell 如果使用双引号，会尝试将 PHP 变量当作 shell 变量进行插值。其作用类似于`-a`交互模式，不同之处在于每行代码执行后，PHP 的“状态”会被清除。这意味着提供的代码行被视为要执行的整个脚本，执行完毕后即终止。示例如下：

```
1 ~$ php -r 'echo (2+2)."\n";'
```

这将打印出`4`并换行。请注意，该行必须是语法正确的 PHP，所以不要漏掉结尾的分号！我们将在本章后面的“调用 PHP 脚本的多种方法”一节中再次讨论`-r`。

* `-B` *或* `--process-begin`

  `-R` *或* `--process-code`

  `-F` *或* `--process-file`

  `-E` *或* `--process-end`

这四个参数允许你指定在 PHP 处理来自`STDIN`的输入之前、期间和之后执行的 PHP 代码。`-B`指定在处理输入之前要执行的一行代码，`-R`指定对每一行输入要执行的一行代码，而`-F`指定对每一行要执行的一个 PHP 文件。最后，`-E`在处理过程结束时执行一行代码。在`-R`和`-F`中，有两个特殊变量可用：`$argn`（正在处理的行的文本）和`$argi`（正在处理的行的编号）。这在直接在 shell 脚本中使用 PHP 时非常有用。例如，要打印带有行号的文本文件，你可以执行如下操作：

```
1 ~$ cat my_text_file.txt | php -B 'echo "Lets add line numbers...\n";'
2       -R 'echo "$argi: $argn\n";' -E 'echo "That''s the end folks\n";'
```

这将输出类似以下内容：

```
1 Lets add line numbers...
2 1: Lorem ipsum dolor sit amet, consectetur adipisicing elit, sed do 
3 2: eiusmod tempor incididunt ut labore et dolore magna aliqua. Ut enim ad
4 3: minim veniam, quis nostrud exercitation ullamco laboris nisi ut aliquip
5 4: ex ea commodo consequat. Duis aute irure dolor in reprehenderit in 
6 That's the end folks
```

* `-S` *或* `--server`

这将启动内置的开发 Web 服务器，它类似于 Apache 或 Nginx 的精简版本，可用于在本地测试或运行单个网站实例。我们将在第 5 章中深入探讨这一点，届时将研究如何使用它来创建基于 HTML 的本地应用。

* `-s` *或* `--syntax-highlight`

这将输出 PHP 脚本的 HTML 版本，并带有彩色语法高亮。PHP 脚本不会被执行或验证，只是被“美化”。美化后的 HTML 会打印到`STDOUT`，在仔细检查代码以查找错误、问题和优化时非常有用。此选项仅适用于文件中的 PHP 代码，不适用于通过`-r`选项提供的代码。大多数现代集成开发环境和代码编辑器默认提供语法高亮，但如果你只能通过命令行访问机器，并且使用的编辑器不支持语法高亮，那么此选项会很有用。在这种情况下，使用`-s`创建脚本的彩色版本，然后下载它，或者如果机器安装了 Web 服务器，则通过 Web 浏览器查看。

* `-v` *或* `--version`

这将输出 PHP 版本信息。这些信息也可以在以上`-i`选项的输出中找到。在假定特定格式时要小心，某些软件包仓库（例如 Ubuntu）会在版本字符串中包含其名称和自身的构建编号，因此不要仅仅为了查找任何数字而进行过滤。

* `-w` *或* `--strip`

这会输出源代码的内容，并去除所有不必要的空白和注释。此选项只能用于代码文件（不能用于`-r`提供的代码行），并且不能与上面的语法高亮选项同时使用。它用于“压缩”文件，即减小文件大小。与普遍看法相反，这不会加速大多数脚本的执行，解析注释和空白的开销是极其微不足道的。即使保留了“完整”代码的副本，你也应警惕支持与调试问题，因为错误报告中的行号在原始版本和剥离版本之间将不再匹配。它也不会压缩像变量名这样的标识符，因此不能用于混淆代码。如今，使用此选项的理由已经非常少。为了分发而减小文件大小，使用适当的压缩（例如添加到 zip 文件）通常是更好的方法。

* `-z` *或* `--zend-extension`

此选项指定在脚本运行前要加载的 Zend 扩展的文件名/路径。这允许动态加载扩展，如果扩展总是需要加载，也可以在`php.ini`文件中指定。
