# 7. 什么是设计模式？为何使用它们？

我们作为程序员遇到的大多数问题，社区中的其他人已经处理过无数次。设计模式可以为我们提供挖掘这些智慧的手段。一旦模式成为通用术语，它就能丰富我们的语言，使分享设计思想及其后果变得容易。设计模式就是提炼常见问题、定义经过验证的解决方案并描述可能的结果。许多书籍和文章专注于计算机语言的细节（如可用函数、类和方法等）。而模式目录则侧重于如何从这些基础知识（“是什么”）出发，进而理解项目中的问题与潜在解决方案（“为什么”和“如何做”）。

在本章中，我将向您介绍设计模式，并探讨它们广受欢迎的一些原因。

本章将涵盖以下内容：

*   模式基础：什么是设计模式？
*   模式结构：设计模式的关键要素是什么？
*   模式优势：为什么模式值得你花时间学习？

## 什么是设计模式？

> 在软件世界中，模式是组织集体记忆的具体体现。—— Grady Booch，《Core J2EE Patterns》

> [模式是] 在特定背景下对问题的解决方案。——“四人帮”，《设计模式：可复用面向对象软件的基础》

正如这些引文所暗示的，设计模式是一种经过分析的问题，并附有其经过良好实践的解决方案。

问题往往会重复出现，作为 Web 程序员，我们必须一次又一次地解决它们。我们该如何处理传入的请求？如何将这些数据转换为系统的指令？我们应该如何获取数据？呈现结果？随着时间的推移，我们会以不同程度的优雅来回答这些问题，并逐渐形成一套非正式的技术体系，并在项目中反复使用。这些技术就是设计模式。

设计模式记录并形式化了这些问题和解决方案，使来之不易的经验能够为更广泛的编程社区所用。模式本质上是（或者说应该是）自下而上的，而非自上而下的。它们根植于实践，而非理论。这并不意味着设计模式缺乏强大的理论元素（我们将在下一章中看到），但模式是基于真实程序员所使用的真实世界技术的。著名的模式孵化者马丁·福勒曾说，他是发现模式，而非发明模式。因此，许多模式会让你产生似曾相识的感觉，因为你认出了自己曾使用过的技巧。

模式目录并非菜谱。菜谱可以照本宣科；代码可以被复制并稍作修改后插入项目中。你甚至不必完全理解菜谱中使用的所有代码。设计模式记录的是解决特定问题的方法。实现细节可能因更广泛的背景而千差万别，这些背景可能包括你使用的编程语言、应用程序的性质、项目的规模以及问题的具体细节。

举例来说，假设你的项目需要一个模板系统。给定一个模板文件的名字，你必须解析它，并构建一个对象树来表示你遇到的标签。

你从一个默认解析器开始，它扫描文本中的触发标记。当找到匹配项时，它会将搜索任务移交给另一个解析器对象，这个解析器专门用于读取标签的内部结构。这个过程会继续检查模板数据，直到失败、完成或找到另一个触发器。如果找到触发器，它也必须将任务移交给一个专门的对象——也许是一个参数解析器。所有这些组件共同构成了所谓的递归下降解析器。

那么，你的参与者包括：一个 `MainParser`、一个 `TagParser` 和一个 `ArgumentParser`。你创建一个 `ParserFactory` 类来创建并返回这些对象。

当然，事情没那么简单。项目后期，你被告知必须支持模板中的多种语法。现在，你需要根据语法创建一组并行的解析器：一个 `OtherTagParser`、一个 `OtherArgumentParser` 等等。

这就是你的问题：你需要根据情况生成一组不同的对象，并且你希望系统其他组件对此基本上是透明的。恰好，“四人帮”在其书中的“抽象工厂”模式总结页中定义了以下问题：“提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。”

这非常契合。正是我们问题的性质决定并塑造了我们对这个模式的使用。解决方案也绝非简单的复制粘贴，正如你将在第 9 章（我将在其中介绍抽象工厂）中看到的那样。

为模式命名这一行为本身就很有价值；它有助于形成一种通用词汇表，这种词汇表在更古老的手艺和职业中早已自然而然地产生。这种速记法极大地促进了协作设计，因为各种替代方案及其不同后果可以得到权衡和测试。例如，当你讨论不同的解析器系列时，你可以简单地告诉同事，系统使用抽象工厂模式创建每组对象。他们会睿智地点点头，要么立刻恍然大悟，要么在心里记下稍后去查阅。关键在于，这组概念和后果有了一个名称，这构成了一个有用的速记法，我将在本章后面说明。

最后，根据国际法，如果不引用克里斯托弗·亚历山大，就撰写关于模式的文章是非法的。他是一位建筑学学者，其工作深刻影响了最初的面向对象模式倡导者。他在《模式语言》（牛津大学出版社，1977）中说道：

> 每一种模式描述了一个在我们环境中反复出现的问题，然后描述了该问题解决方案的核心，这样你就可以无数次地使用这个解决方案，但绝不以相同的方式做两次。

值得注意的是，这个定义（适用于建筑问题和解决方案）从问题及其更广阔的背景开始，然后才转向解决方案。近年来，有一些批评认为设计模式被过度使用了，尤其是在经验不足的程序员手中。这通常表明，在没有问题和背景的情况下，解决方案被应用了。模式不仅仅是特定组织方式的类和对象以特定方式协作。模式的结构旨在定义解决方案的应用条件，并讨论解决方案的效果。

在本书中，我将关注模式领域中一个特别有影响力的流派：“四人帮”在《设计模式：可复用面向对象软件的基础》（Addison-Wesley Professional，1995）中描述的形式。它专注于面向对象软件开发中的模式，并记录了一些存在于大多数现代面向对象项目中的经典模式。

“四人帮”这本书之所以重要，不仅因为它记录了关键的模式，还因为它描述了为这些模式提供信息和动力的设计原则。我们将在下一章中探讨其中一些原则。

> **注**
>
> “四人帮”以及本书中描述的模式实际上是模式语言的实例。模式语言是问题和解决方案的目录，它们被组织在一起，彼此互补，形成一个相互关联的整体。还有其他问题领域的模式语言，例如视觉设计、项目管理（当然还有建筑）。当我在本书中讨论设计模式时，我指的是面向对象软件开发中的问题和解决方案。

### 设计模式概览

从本质上讲，设计模式由四个部分组成：名称、问题、解决方案和后果。

#### 命名

命名至关重要。它们丰富了程序员的语言；短短几个词就能代表相当复杂的问题和解决方案。

命名必须在简洁与描述性之间取得平衡。四人帮（Gang of Four）声称：“找出好名字一直是开发我们目录时最困难的一部分之一。”马丁·福勒（Martin Fowler）对此表示赞同：“模式名称至关重要，因为模式的其中一个目的是创造一种词汇表，让开发者能够更有效地沟通”（*《企业应用架构模式》*，Addison-Wesley Professional，2002 年）。

在*《企业应用架构模式》*中，马丁·福勒完善了一种我最初在迪帕克·阿卢尔（Deepak Alur）、丹·马尔克斯（Dan Malks）和约翰·克鲁皮（John Crupi）合著的*《Core J2EE Patterns》*（Prentice Hall，2001 年）中遇到的数据库访问模式。福勒定义了两种模式来描述对旧有模式的专门化。他方法的逻辑显然正确（一种新模式对领域对象建模，而另一种对数据库表建模，这是早期作品中模糊不清的区别）。尽管如此，我很难训练自己用新模式的术语进行思考。在设计会议和文档中，我使用原模式名称如此之久，以至于它已经成为了我语言的一部分。

### 问题

无论解决方案多么优雅（有些确实非常优雅），问题及其上下文都是模式的根基。识别问题比应用模式目录中的任何一个解决方案都要困难。这也是某些模式解决方案可能被误用或滥用的原因之一。

模式会极其细致地描述问题空间。问题会先被简要描述，然后结合上下文，通常附带一个典型示例和一张或多张图表。问题被分解为其具体细节和多种表现形式。任何有助于识别问题的警示信号都会被描述出来。

#### 解决方案

解决方案起初会与问题一同被概述。它也会被详细描述，通常使用 UML 类图和交互图。模式通常会包含一个代码示例。

尽管可能会呈现代码，但解决方案绝非可复制粘贴之物。模式描述的是解决问题的一种方法。其实现可能有数百种细微差别。想想播种粮食作物的说明。如果你盲目地遵循一系列步骤，到了收获季节很可能就会挨饿。更有用的是一种基于模式的方法，它涵盖了可能适用的各种条件。解决问题的基本方案（让作物生长）总是相同的（准备土壤、播种、灌溉、收获作物），但你实际采取的步骤将取决于各种因素，例如你的土壤类型、地理位置、土地朝向、当地害虫等等。

马丁·福勒将模式中的解决方案称为“半成品”。也就是说，程序员必须理解概念并自行完成它。

### 后果

你所做的每一个设计决策都会产生更广泛的影响。当然，这应包括令人满意地解决所讨论的问题。一个解决方案一旦被部署，可能非常适合与其他模式协同工作。也可能存在需要注意的风险。

## 四人帮（Gang of Four）格式

在我写作时，面前的书桌上放着五本模式目录。快速浏览每本目录中的模式，确认它们没有使用相同的结构。有些是正式的；有些是细粒度的，包含许多子章节；另一些则是散漫的讨论式。


# 有几种定义良好的模式结构

有几种定义良好的模式结构，包括克里斯托弗·亚历山大（Christopher Alexander）开发的原始形式（亚历山大形式），以及波特兰模式知识库（Portland Pattern Repository）偏爱的叙述式方法（波特兰形式）。由于四人帮的书极具影响力，并且我们将涵盖他们描述的许多模式，让我们检视一下他们在模式中包含的几个部分：

- **意图（Intent）**：对模式目的的简要陈述。你应该能够一眼看出模式的要点。

- **动机（Motivation）**：描述的问题，通常以一个典型场景的形式呈现。这种铁事式的方法有助于使模式易于理解。

- **适用性（Applicability）**：审视你可能应用该模式的不同情景。虽然动机描述了一个典型问题，但本节定义了特定情景，并权衡了解决方案在每个情景中的优点。

- **结构/交互（Structure/Interaction）**：这些部分可能包含描述解决方案中类与对象间关系的 UML 类图和交互图。

- **实现（Implementation）**：本节着眼于解决方案的细节。它审视了应用该技术时可能出现的任何问题，并提供了部署技巧。

- **示例代码（Sample Code）**：我总是直接跳到这一部分。我发现一个简单的代码示例往往能提供进入一个模式的途径。示例通常被精简至最基本部分，以揭示解决方案的本质。它可以是任何面向对象语言。当然，在本书中，它将始终是 PHP。

- **已知应用（Known Uses）**：这些描述出现该模式（问题、上下文和解决方案）的真实系统。有人说，要使一个模式真正有效，它必须至少在三个公开可用的环境中被发现。这有时被称为“三条原则”。

- **相关模式（Related Patterns）**：一些模式隐含着其他模式。在应用一个解决方案时，你可以创建另一个解决方案变得有用的上下文。本节审视这些协同效应。它还可能讨论与问题或解决方案有相似之处的模式，以及任何先例（即，在其他地方定义的、当前模式所依赖的模式）。

## 为什么使用设计模式？

那么模式能带来什么好处呢？既然模式是定义好的问题和描述好的解决方案，答案应该是显而易见的。模式可以帮助你解决常见问题。当然，模式的意义不止于此。

### 设计模式定义一个问题

有多少次你到达项目的一个阶段，却发现无路可进？在重新开始之前，你很可能需要先退后几步。

通过定义常见问题，模式可以帮助你改进设计。有时，解决问题第一步就是认识到你遇到了问题。

### 设计模式定义一个解决方案

在定义并识别出问题（并确保它是正确的问题）之后，模式能为你提供一个解决方案，并附带对其使用后果的分析。尽管模式并不能免除你考虑设计决策影响的责任，但至少你可以确信你正在使用一种经过检验的技术。

### 设计模式与语言无关

模式用面向对象的术语定义对象和解决方案。这意味着许多模式同等适用于多种语言。当我刚开始使用模式时，我阅读的是 C++和 Smalltalk 的代码示例，然后在我的 Java 解决方案中进行部署。其他人则会根据对模式的适用性或后果进行调整，但依然保持有效。无论哪种情况，模式都可以在你切换语言时提供帮助。同样，一个基于良好面向对象设计原则构建的应用程序，在不同语言之间移植也相对容易（尽管总是存在一些需要解决的问题）。

### 模式定义了一种词汇表

通过为开发者提供技术的名称，模式使沟通更加丰富。想象一次设计会议。我已经描述了我的 `Abstract Factory` 解决方案，现在需要说明管理系统编译数据的策略。我向鲍勃描述我的计划：

-   我：我正考虑使用 `Composite`。

-   鲍勃：我觉得你还没想清楚。

好吧，鲍勃不同意我的看法。他向来如此。但他明白我在说什么，因此也知道我的想法为什么糟糕。让我们在没有设计词汇的情况下重新演绎这个场景：

-   我：我打算使用一个共享相同类型的对象树。该类型的接口将提供用于添加自身类型的子对象的方法。通过这种方式，我们可以在运行时构建出实现对象的复杂组合。

-   鲍勃：啊？

模式，或者说它们所描述的技术，往往是相互协作的。例如，`Composite` 模式倾向于与 `Visitor` 模式协同工作：

-   我：然后我们可以使用 `Visitor` 来汇总数据。

-   鲍勃：你没抓住重点。

忽略鲍勃。我就不描述这种没有模式时的曲折版本了；我将在第 10 章 介绍 `Composite`，在第 11 章 介绍 `Visitor`。

关键在于，即使没有模式语言，我们仍然会使用这些技术。它们先于其命名和组织而存在。即使模式不存在，它们也终究会自行演化。任何被充分使用的工具最终都会获得一个名称。

### 模式是经过考验的

因此，如果模式记录了良好实践，那么命名是否就是模式目录中唯一真正原创的东西？从某种意义上说，这似乎是事实。模式代表了面向对象环境中的最佳实践。对于某些经验丰富的程序员来说，这似乎只是对显而易见之事的重新包装。对于其他人来说，模式提供了一条接触问题和解决方案的途径，否则我们只能通过艰难的方式去发现它们。

模式让设计变得平易近人。随着越来越多特定领域的模式目录出现，即使是经验丰富的人在涉足其领域的新方面时也能从中受益。例如，GUI 程序员可以快速了解企业编程中的常见问题和解决方案。Web 程序员可以快速制定策略，避免平板电脑和智能手机项目中潜藏的陷阱。

### 模式是为协作而设计的

就其本质而言，模式应该是可生成和可组合的。这意味着你应该能够应用一个模式，从而创造出适合应用另一个模式的条件。换句话说，使用一个模式时，你可能会发现其他门路为你敞开。

模式目录的设计通常都会考虑到这种协作，而模式组合的可能性也总是记录在模式本身。

### 设计模式提升良好设计

设计模式展示并应用了面向对象设计的原则。因此，研究设计模式不仅仅能让你获得特定上下文中的解决方案，你还能对组合对象和类以实现目标的方式产生新的视角。

### 设计模式被流行框架所使用

本书主要讨论从头开始设计。这里涵盖的模式和原则应能使你根据项目需求设计自己的核心框架。然而，懒惰也是一种美德，你或许希望使用像 `Zend`、`Laravel` 或 `Symfony` 这样的框架（或者你可能会接手已经使用这些框架的代码）。深入理解核心设计模式将有助于你在使用这些框架 API 时更好地开展工作。

## PHP 与设计模式

本章几乎没有特定于 `PHP` 的内容，这在某种程度上是这个话题的特点。许多模式适用于多种支持对象的语言，几乎没有或根本不存在实现问题。

当然，情况并非总是如此。有些企业级模式在那些应用进程能在服务器请求之间持续运行的语言中效果很好。`PHP` 并非如此。每次请求都会启动一次新的脚本执行。这意味着某些模式需要更谨慎地对待。例如，`Front Controller` 通常需要一些显著初始化时间。如果在应用程序启动时只进行一次初始化，这没问题，但每次请求都必须初始化时，就成问题了。这并非说我们不能使用该模式；我过去曾部署过它并取得了非常好的效果。我们只是必须确保在讨论该模式时考虑到与 PHP 相关的问题。`PHP` 构成了本书所讨论所有模式的上下文。

我在本节前面提到了支持对象的语言。你可以完全不定义任何类就用 `PHP` 编写代码。然而，除少数几个显著的例外情况外，对象和面向对象设计是大多数 `PHP` 项目和库的核心。

## 总结

在本章中，我介绍了设计模式，向您展示了它们的结构（使用"四人帮"的形式），并提出了一些在脚本中使用设计模式的原因。

重要的是要记住，设计模式不是像组件一样拼凑起来就能构建项目的即插即用解决方案。它们是针对常见问题的建议性方法。这些解决方案体现了一些关键的设计原则。这些原则正是我们将在下一章中要探讨的内容。
