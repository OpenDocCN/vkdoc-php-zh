# 2. 什么是单元测试？

![A337704_2_En_2_Figa_HTML.jpg](img/A337704_2_En_2_Figa_HTML.jpg)

把手给我，宝贝

单元测试允许你对软件进行隔离测试。通常情况下，单元测试规模小、运行快，只测试一小段代码——即一个单元——例如一个函数。单元测试本身也是代码，旨在测试其他代码。

它有助于保持代码简洁、易于理解和阅读。同时，它也有助于编写更易于维护的代码。最终，这有助于创建缺陷更少的软件。单元测试将测试过程自动化。

单独依靠单元测试并不能保证应用程序完美无缺、毫无缺陷，但它们确实能带来很大帮助。

## 从手动测试到单元测试

当你编写代码时，必须确保它能完成预期工作。由于我们讨论的是 PHP 测试，新手应在浏览器中进行测试。编写代码；打开浏览器；输入不同数据；检查输出是否符合预期。听起来很熟悉？这被称为手动测试，通过这种方式，我们测试的是功能，而非代码本身。

下一步是建立一个自动化测试系统，这样你就不必在浏览器中手动点击各处来检查软件了。有许多工具可以完成这项工作。两种主要方法是测试驱动开发（TDD）和行为驱动开发（BDD）。

简而言之，TDD 要求在编写代码之前先编写测试。这种方法依赖于单元测试。

由于单元测试针对的是小段代码，编写起来很容易，并且你的代码也很容易被它们覆盖。

在 BDD 中，任何单元的测试都应基于该单元的期望行为来指定。

## 争论

关于反对编写单元测试，存在一些常见论点。我想通过回应这些论点，并分享我在这些问题上的个人经验，来强调单元测试的优势。然后，你可以亲自尝试单元测试，并做出自己的判断。

### 论点 #1：不可能测试所有变体

是的，确实如此。你无法测试所有可能的变体。但至少单元测试能显示你的测试覆盖了哪些变体。如果你能确定应用程序因某个未覆盖的变体而失败，只需扩展测试即可。

例如，如果我的测试能在正整数上成功运行，那么日后必要时，我可以将其扩展为测试 0、负整数或浮点数。

### 论点 #2：编写测试太耗时

编写（以及维护）额外的代码确实需要时间。这显而易见。但编写本身只是众多耗时因素之一。还有哪些？花费在理解自己三、四个月前所写代码上的时间；花费在理解他人代码上的时间；花费在寻找包含 bug 的特定代码片段上的时间；每当第三方代码变更时，花费在重新审查代码上的时间。当我们把所有这些事情花费的时间加起来，就会发现单元测试似乎能够节省时间。节省时间意味着节省成本，但要小心。单元测试**可能**节省时间，并非**绝对**会节省时间。你必须正确使用测试。

学习使用单元测试本身也需要时间。但我认为学习曲线很短暂，并且即使在你第一次使用时，也会有所回报。

### 论点 #3：编写测试很难

编写**测试**很容易。编写**好的测试**才难。但这只需要一点练习。

有一个例外。为面条式代码，或做很多事情的函数编写测试，确实很难。因此，当你遇到困难的测试时，应意识到这可能表明你的代码需要重构。

### 论点 #4：我不需要测试；我了解自己的代码

大约 99.9% 的情况下，你并非独自编码。我们经常使用第三方代码。而且，随着时间的推移，我们往往会忘记或改变自己对编码方式的思考。也许当初编写代码时思路很清晰，但一段时间后，我们可能需要花费几分钟（甚至几小时）来理解自己当时做了什么。

很多时候，我会为自己仅五六行的辅助函数编写测试。当你的代码运行在不同的服务器、不同的 PHP 版本上时，这些函数测试起来要简单得多。测试不同的输入——甚至包括那些根本不应该出现的数据——比如除零错误，也更简单。

### 论点 #5：这纯粹是浪费时间

这个论点通常来自那些刚开始编写单元测试，且心中抱有论点 #4 想法的 Web 开发者。他们常说：“我写了一些单元测试，但运行时一个 bug 都没发现。我觉得这是在浪费时间。”

好吧，现在你有了测试。这意味着你手头有了一项资产，它可以在你进行某些修改或重构后，帮助你检查代码的相同部分。至少，为此你无需再回到手动测试。编写更多的测试，三个月后再回到你的代码，看看有什么不同。编写代码和维护代码是两件不同的事。有了单元测试，你在维护代码上浪费的时间会比没有它们时少得多。

### 论点 #6：测试本身可能也有 bug

是的。任何代码都可能存在 bug。但是单元测试通常很短，因此更容易编写成无 bug 的版本。完全有可能编写出即使本应失败的测试也能通过的通过测试。再次强调，解决方案在于练习。

永远不要忘记：测试能证明错误的存在，但不能证明错误不存在。



### 论点 #7：开发过程会破坏测试

随着你在开发周期中不断添加新层，你同样需要维护你的测试。当代码发生变化时，相应的测试也应随之改变；否则，测试将会失败。不过，既然你修改了代码，你就应该知道如何修改测试。

实际上，这正是拥有测试的优势之一。当我重构代码时，测试应该能够成功运行。如果我修改了代码的签名或返回值，测试会失败，或者也可能成功运行。如果失败，你就能看到代码在哪些地方可能会产生问题或 bug。因此，它促进了安全的修改。

## 我们为什么要编写测试？

测试能显著缩短测试时间，这最终会节省成本。测试有助于编写高质量的代码，并且是重构过程中的重要资产。它们将改变你对代码的思考方式。

### 测试功能

当我们首次编写代码时，应该检查它是否真的实现了我们想要的功能。我们所有人都会这样做——无论有没有单元测试。实际上，`var_dump` 及其同类工具也算是一种测试。只是它们无法满足测试的真正目的。

一段时间后，你必然需要对旧代码进行修改，要么是因为存在 bug，要么仅仅是因为你想优化之前的工作。在这种情况下，你也需要一些手段来测试代码是否依然满足你的需求。手动测试既枯燥、缓慢又低效。

与任何其他测试系统相比，手动测试耗时都多得多，至少在你认真细致地操作时是如此。同一个方法可能在应用程序的许多地方被调用，你需要测试所有地方，以确保你的应用程序按预期工作。除此之外，你还应该列出可能的输入列表，例如“我应该用负数、零、字母、特殊字符来测试它。” 也许在部署到生产环境之前，你忽略了测试所有情况。不，这种情况绝对不会发生！

### 重构

重命名变量或函数可以提高代码的可读性。但你必须注意，重命名不会引入新的 bug 或冲突。修改方法的签名或返回值也是如此。测试能为你提供帮助。

有时候，代码中会存在一些与代码目标毫无关联，或者根本不做任何事情的片段。显然，你想要移除这些代码，但之后你必须检查代码的主体部分是否仍然能正常工作。如果你的代码有测试覆盖，你就可以安全地移除那些不需要的代码片段，因为如果你移除了有用的部分，测试就会失败。

在另一些时候，你可能知道自己的代码有问题，但不想花时间去修复。于是，你只是在代码中留下一段注释，比如 `// TODO: 如果 $num = 0 将会出错`。而一个失败的测试则要清晰得多。

有句名言：“如果没坏，就别修它。” 你可能会问：我辛苦写出来的代码运行得好好的，你在说什么呢？但简单来说，这句话对于软件开发并不适用。重构是开发中非常重要的一部分。通过成功地重构代码，你最终会得到和没有重构时相同的结果，但你的代码将变得可复用、可维护、更快、更整洁，并且质量更高。

### 获取快速反馈

快速反馈非常重要。开发人员没有时间点击每一个链接，以所有可能的顺序按每一个按钮，多次按浏览器的“返回”按钮，用各种数据（甚至是恶意数据）填写表单，或者部署新代码然后等待客户反馈。

我认为 Firebug ( [`getfirebug.com`](https://getfirebug.com) ) 是一个很好的例子。我打开 HTML 选项卡并更改一些内容，我立刻就能在浏览器中看到结果。

还有其他语言的工具，它们通过在每一个分号后或每次保存文件时实际运行测试来提供即时反馈。这些都是极其强大的开发工具。虽然 PHP 中没有这样的工具，但无论如何，测试是实现这一目标的最佳工具。

### 高质量的代码

测试是在隔离环境中运行的。因此，通过运行一个测试，你正在验证一个不依赖于其他代码段的特定代码片段。但这很容易违反单一职责原则。测试为我们的代码编写过程增加了一个视角。更多内容可参见第 3 章。

### 发挥大脑的最佳状态

在压力之下，即使是最聪明的人（是的，甚至包括程序员）也会犯愚蠢的错误。

你曾经写过像 `if(3 < 5)` 这样的代码吗？从来没有？我保证，你会的。即使是最微不足道的代码或代码变更，你也很可能会破坏某些东西。你是否总是放松？你是否总是有足够的时间编写代码？你的老板是否总是开心？你的客户是否总是耐心？你是否总是有明确的规格说明？你的孩子是否整夜安睡？如果答案是否定的，你就会犯愚蠢的错误。测试能帮助你以一种简单的方式检查你的代码。

### 节省时间和金钱

正如我之前提到的，无论通过何种方式，你终究会测试你的代码。但你可以选择一种更有效的方法。你可以选择一种让你花更少时间修复 bug 和重构，而花更多时间实际编写代码的方法。节省时间很有趣——你总是可以把这些时间用来追你最喜欢的下一集电视剧——而节省时间也就意味着节省金钱。

## 总结

在本章中，你学习了我们为什么要测试，什么是单元测试，以及反对它的论点是什么。由此，你了解到你可以节省维护代码的时间——因为几个月后的代码对你来说可能就像别人写的一样陌生——以及测试如何帮助你发挥大脑的最佳状态。
