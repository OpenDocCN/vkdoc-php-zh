# 8.5 在整個應用程式前加上 `CloudFront`

我們的應用程式內容是動態的。這使得在其前方放置一個 CDN（例如 `CloudFront`）變得困難（但並非不可能，我們將看到）。使用 `CloudFront` 會帶來一個問題——我們如何防止用戶看到過時的內容？

事實上，大多數 CDN 的處理方式與我們處理本地快取的方式非常相似。我們可以單純地為內容設定一個最大到期日期。

在 `CloudFront` 中，如果你查看你的分發行為，你會看到一個「行為」頁籤。點擊它，你會看到一個單一的預設行為。這些行為允許你為伺服器上不同的內容路徑設定不同的設定。以這個例子來說，我們只需要編輯已經存在的那個行為。選取目前的行為，然後點擊「編輯」。

如果你向下捲動，你會看到一組 TTL（存留時間）值，單位是秒。要更改這些值，首先將「物件快取」設定從「使用來源快取標頭」改為「自訂」。如果你不希望內容超過 5 秒鐘，請將「最大 TTL」設定為 5。這意味著快取只會在重新查詢內容前，將內容保留 5 秒鐘。因此，內容可能會有一點點過時，但不會太嚴重。

這引出了另一個問題——CDN 將如何處理向伺服器發送表單？這也可以透過行為設定來處理。在「允許的 HTTP 方法」下，選擇包含所有 HTTP 方法的選項。CDN 只會快取 `GET` 和 `HEAD` 請求，但會將所有其他請求直接轉發到你的伺服器。

此外，預設情況下，`CloudFront` 不會將 Cookie、HTTP 標頭或查詢字串轉發到伺服器。這是為了減少它必須發出的原始請求數量，以及它必須快取的物件數量。然而，在我們的應用程式中，你看到的內容是基於查詢字串的。因此，你需要將「查詢字串轉發和快取」設定為「轉發所有，基於全部進行快取」，才能使應用程式正常運作。

設定完成後，點擊「是，編輯」以儲存行為中的變更。

你現在可以前往你的 `CloudFront` 分發行為的 URL，並像使用一般網站一樣使用它。你也可以按照第 8.4 節的說明，讓用戶透過你網站的專屬主機名稱來存取它。

在你對這種應用程式設定過於熱衷之前，請記住，所有快取設定都是一種平衡。如果你為快取使用了太多鍵（例如，路徑、標頭、查詢字串、Cookie），那麼你的大部分內容都將被傳遞到伺服器（即不被快取），而你將無法獲得效能提升。事實上，這只會增加開銷和成本，因為你必須同時支付快取的頻寬和伺服器的頻寬。然而，如果快取設定過於寬鬆，用戶將看到過時的數據，或者更糟，看到其他人的數據！例如，如果你的內容是根據 Cookie 進行個人化的，但你沒有將 Cookie 作為快取鍵的一部分，那麼如果 Jim Bob 請求一個頁面並被放入快取，當 Jane Doe 請求同一個頁面時，她將看到 Jim Bob 的頁面！

那麼，擁有高度個人化網路應用程式的人如何更好地利用 CDN 呢？答案是將你的應用程式由內而外翻轉。
