# 第四部分：解耦式 Drupal 生态系统

解耦式 Drupal 生态系统

在第三部分中，我们不再描述 Web 服务，而是直接实现用于消费和操作 Drupal 内容的请求。这为我们开始审视如何用不同技术实际开发消费者应用程序奠定了合适的基础。然而，在转向该主题之前，我们将在本章中讨论蓬勃发展的解耦式 Drupal 生态系统，其目标是简化将 Drupal 用作内容服务的过程，并加速消费者应用程序的开发。

当 Drupal 8 发布时，许多 Drupal 开发者尚未设想将 Drupal 解耦。因此，通过典型设置流程生成的标准 Drupal 8 安装，其功能常常过于臃肿，无法高效地用作解耦式 Drupal 后端。虽然站点维护人员可以禁用不需要的模块，但这一过程可能相当繁琐。然而，得益于 Drupal 发行版（即拥有不同模块和功能集的不同 Drupal 变体）的使用，解耦式 Drupal 实践者已经开始采用 API 优先的发行版。这些 Drupal 版本针对解耦式 Drupal 用例进行了优化，具备预先启用的 Web 服务、预配置的设置以及增强的用户界面等功能。

得益于围绕 Contenta 的发展以及 Reservoir 等项目引起的兴趣，API 优先的发行版在未来几年内仍将是一个处于重大转型期的领域。当谈到 API 优先发行版的理想功能集时，许多架构师尚未形成完整的愿望清单，但不同的方法表明了浓厚的兴趣。如果有朝一日，一个专门针对解耦式 Drupal 用例的安装配置文件进入 Drupal 核心，它必定会从 Contenta、Reservoir 及其他项目中汲取大量灵感。

此外，在无头 CMS 取得成功的所有主要原因中，最重要的或许就是软件开发工具包（SDK）和参考构建的普及，它们促进了开发者对消费者应用程序的开发。这些软件项目为非 Drupal 实践者提供了一种“说”Drupal 语言并能快速上手构建其消费者的方法。虽然内容即服务平台将 SDK 和参考构建作为免费开源软件提供，但其平台的其余部分仍处于保密状态。近年来，Drupal 坚定地采取了从头到尾（无论是单体式还是解耦式）均开源 CMS 的立场。Waterwheel 和 Contenta 生态系统中新出现的 SDK 和参考构建都表明市场对便捷消费 Drupal 内容有着健康的需求。

借助 SDK、参考构建以及以 `Contenta.js` 形式存在的中间件层，任何技术背景的开发者，无论是否接触过 Drupal，都能像使用任何 API 一样轻松消费和操作 Drupal 内容。像 `Waterwheel.js` 这样的 SDK 旨在用于通用场景，而更为“演示就绪”的参考构建（如 Waterwheel 的 Ember 和 React 应用程序以及 Contenta 的示例应用程序）可以给评估者带来更大的安心感。
