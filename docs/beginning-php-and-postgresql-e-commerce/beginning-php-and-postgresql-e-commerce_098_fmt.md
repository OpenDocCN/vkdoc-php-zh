# 更多项目管理方法

除了目前所述的方法外，你还能采用更多项目管理方法。由于没有哪种单一方法是最优的，一名优秀的项目经理必须从理论上对所有方法都有所了解，以便为当前项目选择最佳方法。为项目选错策略可能会导致失败，因此项目经理在决定如何应对特定项目之前，需要仔细权衡所有选项。像这样的策略永远行不通：“好吧，我们要建一个电商网站。这次我们用 XP（极限编程）来做，下次也许再用螺旋模型！”

在许多情况下，最好混合使用多种方法来适应你的项目。例如，如果客户不确定自己想要什么，你可以运用部分 XP 理念，基于用户故事文档在开发过程中与客户紧密协作，再添加一些瀑布模型的步骤，并在过程中进行部分快速应用开发（RAD）。

无论如何，为你的下一个项目牢记其中一些流程至关重要，因为项目管理方式能帮你节省时间、金钱，并减轻压力。

## 理解电子商务项目周期

如果你具备一些管理知识，并且对网页设计有良好的艺术感知力，那么读完这本书后，电子商务项目完全可以“单枪匹马”完成。首先，你需要组织任务，使其按逻辑顺序依次进行。

**理解客户需求**应该不难。客户想要一个能展示和销售一系列产品的在线商店。你需要了解客户希望在网站上销售的产品类型，以及一点未来战略（比如客户现在只卖硬件组件，但未来可能想卖笔记本电脑）。这一点非常重要，因为数据库和整个架构必须从一开始就设计成能够支持未来的变更。你可能还需要了解发货部门是如何组织的，以便优化处理和发货流程。

大多数客户都要求具备美观性和功能性的设计，因此，在大多数情况下，下一步是**创建网页原型**。无论你是自己动手还是雇佣网页设计师，原型应该只是一个网站模板——只有 HTML 代码，用“此处为产品名称”之类的占位符代替实际产品，不需要任何数据库。根据客户的审美偏好，你可能需要构建多个原型，直到双方就设计达成一致。

**设计数据库**，正如我所说，是项目的一个关键阶段。逻辑数据库设计是在需求收集阶段形成的，并与客户达成一致。数据库的逻辑设计描述了需要存储的数据以及不同数据实体之间的关系（例如产品与部门之间的关系），但不包括严格的实现细节，比如用于物理实现多对多关系的关联表。如果你是高级数据库设计师，你可以自行创建最优的物理数据库结构。

有许多工具（例如 `http://www.infogoal.com/dmc/dmcdmd.htm` 上介绍的）可以让你以可视化方式设计数据库。（你也可以通过谷歌搜索“数据建模”找到更多有用的链接。）这些工具在设计关系型数据库结构方面功能非常强大，甚至能生成 SQL 代码来将它们转化为真实的数据库。无论你使用哪种数据库引擎，都应该以可视化方式设计你的表（哪怕只用纸笔），而不是一开始就编写 SQL 查询。

接下来是**实现数据层对象**。这是你开始摆弄数据库的阶段，因为你需要实现数据访问逻辑，以支持应用程序中的其他层。在此过程中，你可能希望用一些虚构的数据填充数据库，作为测试查询的基础。在将查询编写为数据层对象之前，请使用数据库引擎的可视化界面来测试它们，该界面允许执行和调试 SQL 查询。这将让你在调试 SQL 代码时轻松不少，因为所有 SQL 开发者都知道，代码并不总是像你第一次预期的那样工作。

数据层就绪后，你可以继续**构建应用程序的中间层**。在本书中，你学习了一些为网站各个部分实现中间层的技术，但针对你的特定项目，你可能会选择其他技术。

**构建用户界面**显然是下一步。你已经有了一个原型，但它仅适用于设计，因为在创建原型的阶段，你还没有一个功能基础。通常，软件项目中的界面原型是会被丢弃的代码，但在这里，你构建的是 UI 逻辑（最好使用`Smarty`或其他模板引擎），该逻辑根据客户认可的设计生成网站的实际外观。

项目结束时，**最终测试阶段**非常重要。数据库将被填充真实记录，并进行模拟以测试订单处理流程的效率。每个流程在投产前都应进行测试，因此你必须给客户足够的时间来测试网站的每一项功能，进行一些测试订单，并评估发货流程。在这个阶段，任何编程错误都应被暴露出来，供你修正。

代码准备就绪并在本地机器上测试完成后，下一步是**寻找/提供托管解决方案**。也许最好的策略是先将项目托管在专业的服务商处，如果网站获得成功，客户可以投资建立自己的托管解决方案。

## 维护与客户的关系

在理想的项目中，你会将所有可能的条款都写入合同；在交付网站并完成项目后，你希望再也不必听到客户的消息，除非是开发新功能或更改设计——这种情况下，你可以向客户额外收费。

最不想要的情况是客户要求你免费进行修改。如果你在合同上以及提供给客户的管理工具上不够小心，这种情况就可能发生。

例如，许多电商网站的商品目录管理页面做得很差，这对程序员来说是噩梦。通过为客户提供合适的工具和界面，并且最重要的是，描述清楚它们的工作原理（最终提供一份用户手册），就有可能避免这种噩梦。许多程序员不把这个细节当回事，宁愿在商品目录管理页面不完整或难以使用的情况下就上线网站，却不知道后续会有什么麻烦。

如果数据库很复杂，你必须在手册中描述所有字段及其填写方式；如果客户在尝试向数据库提交表单时发生错误，你必须让错误页面尽可能清晰明确。此外，在设计阶段，尽量与那些将使用你提供的工具的人一起工作，并花几个小时亲自指导他们如何使用这些工具。这将为你节省大量电话解释的时间，甚至避免你无偿跑到客户办公室。
