# 8.6 將你的應用程式由內而外翻轉

我們在上一節遇到的問題是，如何向用戶提供高度客製化的頁面，同時仍能充分利用快取。擁有客製化頁面通常意味著它們不可被快取。這個困境可以透過為你的網站採用「由內而外」的頁面架構來解決。

從歷史上看，網路應用程式是透過讓伺服器產生頁面來建立的。當用戶導覽到一個 URL 時，伺服器會擷取一個範本，將其與用戶數據結合，並加入頁面內容以產生最終的 HTML 頁面。然後，這個 HTML 頁面作為一個整體被送回給用戶。

這種架構之所以運作良好，有以下幾個原因：

- 大多數網路應用程式框架都是圍繞這個範例建立的。
- 這種架構與 HTML 的結構方式密切匹配。
- 這種架構易於構想和建置。
- 這種架構所需的實作規劃少得多。
- 這種架構需要的初始開發工作非常少。
- 網路歷史的很大一部分根源於這種架構。
- 這種架構的優化是最後才進行的（即，過早優化是萬惡之源）。

然而，這種架構的問題在於，對於大多數使用案例來說，它在高流量情況下的效率非常低。

另一種組織應用程式的方式是將你的頁面由內而外翻轉，這得益於多年來 Ajax 技術的發展。也就是說，在典型的架構中，伺服器會產生一個包含動態內容的頁面。在新的方式中，伺服器產生的頁面基本上是靜態的，但會回呼伺服器以獲取任何動態內容。

舉例來說，想像一個列出你產品的頁面。在典型的網路框架中，你的伺服器會執行以下任務：

1.  用戶的瀏覽器向伺服器請求一個特定頁面。
2.  伺服器啟動網路應用程式處理引擎（PHP、Ruby on Rails 等）。
3.  應用程式存取資料庫以獲取產品清單。
4.  應用程式為產品清單產生 HTML。
5.  應用程式存取資料庫以獲取用戶的狀態資訊（即，登入資訊、購物車資訊、其他狀態資訊等）。
6.  應用程式產生包含用戶狀態資訊及其在頁面導覽中位置的 HTML 頁首和頁尾。
7.  應用程式從產生的片段拼接出整個 HTML 頁面。
8.  伺服器將最終的 HTML 頁面送回給用戶。
9.  用戶的瀏覽器渲染頁面。

如你所見，這是一個相當複雜的過程，而且在整個頁面在伺服器上渲染完成之前，用戶得不到任何回饋。這導致用戶花費大量時間等待伺服器完成。

然而，如果我們將其由內而外翻轉，我們可以實現更好的優化策略。我們要做的是將頁面作為靜態頁面提供，然後讓用戶自己的瀏覽器負責獲取基於用戶的內容並拼接頁面。

新的順序如下所示：

1.  用戶的瀏覽器向 CDN 請求頁面。
2.  假設它已經被快取，靠近用戶的 CDN 伺服器會立即以頁面內容（即，產品清單）回應。
3.  用戶的瀏覽器載入內容並立即顯示，在任何用戶專屬數據旁顯示一個載入中的旋轉圖示。
4.  頁面執行 JavaScript，向伺服器發送一個請求，以獲取用戶的狀態資訊。
5.  伺服器發回用戶的 JSON 編碼狀態資訊。
6.  頁面渲染網頁的其餘元件。

### CDN 与服务器负载

在此序列中，服务器的负担大大减轻。它不再需要查询产品列表、生成页眉和页脚，或拼接页面。所有这些都发生在用户和 CDN 之间非常快速的事务中。服务器仅负责用户的会话状态（甚至这通常也可以本地化到用户自己的浏览器中！）。这使得大量动态、交互式内容能够在服务器负担很小的情况下实现。

这个概念可以进一步扩展，甚至产品列表也可以在事后生成。API 请求本身也可以被缓存，使用与主网页相同或不同的设置。

如您所见，如果应用程序经过精心设计，CDN 为提升网站速度提供了非常灵活和强大的工具。

这种方法的主要缺点在于实施所需的大量规划和远见。这种方法需要决定一个`Ajax`和动态 HTML 框架，规划一个良好的应用程序架构，开发 API 和 API 身份验证，并规划将要整合所有部分的页面布局。在旧的架构中，您通常可以相当随意地快速拼凑出页面，它们仍然可以工作。当使用“由内而外”的架构时，您必须从一个计划开始，并且需要相当多的前期努力。本书不打算展示“由内而外”页面架构的代码，正是因为它需要大量代码来实现。

无论如何，无论您的应用程序架构如何，大多数应用程序都可以从某种 CDN 解决方案中受益。
