# 14. 好（与坏）实践

到目前为止，本书主要聚焦于编码，并特别关注设计在构建灵活可复用的工具和应用中的作用。然而，开发并不仅限于代码。人们可能会在阅读书籍和完成课程后，对一门语言有了扎实的理解，但在运行和部署项目时仍然会遇到问题。

在本章中，我将超越代码本身，介绍一些构成成功开发过程基础的工具和技术。本章将涵盖以下内容：

- 第三方包：从哪里获取以及何时使用
- 构建：创建和部署包
- 版本控制：为开发过程带来和谐
- 文档：编写易于理解、使用和扩展的代码
- 单元测试：用于自动检测和预防 Bug 的工具
- 标准：为什么有时随大流是好的做法
- `Vagrant`：一个利用虚拟化的工具，使所有开发者无论其硬件或操作系统如何，都能在类似于生产环境的系统中工作
- 持续集成：使用这一实践和工具集来自动化项目构建和测试，并在问题发生时及时收到警报

## 超越代码

当我最初从独立工作转变为加入开发团队时，我惊讶于其他开发者似乎需要了解那么多东西。关于一些看似至关重要的问题，善意的争论无休止地发酵：哪个文本编辑器最好？团队是否应该统一使用一个集成开发环境？我们是否应该强制实施编码标准？我们应该如何测试代码？我们应该在开发过程中记录文档吗？有时，这些问题似乎比代码本身更重要，而我的同事们似乎通过某种神秘的渗透过程，获得了他们对该领域百科全书般的知识。

我之前读过的关于 PHP、Perl 和 Java 的书籍，当然没有在很大程度上偏离代码本身。正如我之前讨论过的，大多数关于编程平台的书籍很少偏离对函数和语法的紧密关注，而涉及代码设计。如果设计都算是题外话，那么你可以肯定，像版本控制和测试这样的更广泛问题就更是鲜少被讨论。这并非批评——如果一本书声称涵盖了某种语言的主要特性，那么它只做这些事也就不足为奇了。

然而，在学习编码的过程中，我发现我忽略了项目日常运作的许多机制。我意识到，其中一些细节对我所参与开发的项目的成败至关重要。在本章及后续章节的详细讨论中，我将超越代码，探索一些可能决定你项目成败的工具和技术。

## 借用轮子

当项目中遇到一个具有挑战性但相对独立的需求时（例如，需要解析特定格式，或使用新协议与远程服务器通信），构建一个满足需求的组件是很有价值的。它也可能是学习技艺的最佳途径之一。在创建包的过程中，你能深入理解问题，并掌握可能具有更广泛应用的新技术。你既是在为项目投资，也是在提升自身技能。通过将功能保留在系统内部，你可以让用户免于下载第三方包。有时，你还能规避棘手的许可问题。当你测试自己设计的组件，并惊奇地发现它竟然能正常工作——完全实现了你承诺的功能时，那种成就感是无可替代的。

当然，这也有其阴暗面。许多包凝聚了数千工时的投入：这是你可能并不具备的资源。你或许可以通过只开发项目所需的功能来解决这个问题，而第三方工具可能还能满足其他众多需求。但问题依然存在：如果已有免费可用的工具，为何还要浪费才华去重新创造它？你是否有时间和资源来开发、测试和调试自己的包？这些时间难道不能更好地用在别处吗？

在重复造轮子这件事上，我算是罪魁祸首之一。拆解问题并发明解决方案，是我们作为程序员工作的核心部分。着手进行一些严肃的架构设计，比编写胶水代码将三四个现有组件拼凑起来更有成就感。每当这种诱惑袭来，我都会提醒自己过往的项目。虽然在我经验中，选择从头构建从未导致项目失败，但我目睹过它吞噬项目进度、扼杀利润。我坐在那里，眼中闪烁着狂热的光芒，构思方案、绘制类图，却未曾注意到，当我沉迷于组件的细节时，全局蓝图早已变得遥不可及。

如今，在规划项目时，我会努力培养一种直觉，判断哪些内容应属于代码库内部，哪些应作为第三方需求处理。例如，你的应用可能需要生成（或读取）RSS 源；你可能需要验证电子邮件地址、自动发送邮件、验证用户身份，或读取标准格式的配置文件。所有这些需求都可以通过外部包来实现。

在本书的先前版本中，我建议使用 PEAR（PHP 扩展与应用库）来获取包。但时代在变，PHP 世界已明确转向 Composer 依赖管理器及其默认仓库 Packagist（[`https://packagist.org`](https://packagist.org)）。由于 Composer 按项目管理包，它不太容易引发可怕的“依赖地狱”综合征（即不同包需要同一库的不兼容版本）。此外，所有行动都已转移至 Composer/Packagist，这意味着你更有可能在那里找到所需内容。更重要的是，许多 PEAR 包也通过 Packagist 提供（[`https://packagist.org/packages/pear/`](https://packagist.org/packages/pear/)）。

因此，一旦你定义了需求，第一站就应该是 Packagist 网站。然后你可以使用 Composer 安装包并管理包依赖。我将在下一章更详细地介绍 Composer。

为了让你了解使用 Composer 和 Packagist 可以获取什么，以下是你能在那里找到的众多包所能实现的一些功能：

- 使用 `pear/cache_lite` 缓存输出
- 使用 `athletic/athletic` 基准测试库测试代码效率
- 使用 `doctrine/dbal` 抽象数据库访问细节
- 使用 `simplepie/simplepie` 提取 RSS 源
- 使用 `pear/mail` 发送带附件的邮件
- 使用 `symfony/config` 解析配置文件格式
- 使用 `jwage/purl` 解析和操作 URL

Packagist 网站提供了强大的搜索功能。你可以在那里找到满足需求的包，或者可能需要使用搜索引擎扩大搜索范围。无论如何，在着手可能重复造轮子之前，你都应该花时间评估现有的包。

你有需求，并且存在满足该需求的包——这不应该是你决策的唯一依据。虽然使用包能节省不必要开发的成本是可取的，但在某些情况下，它可能徒增开销而无实际收益。例如，你的客户需要应用能发送邮件，这并不意味着你应该自动使用 `pear/mail` 包。PHP 提供了相当好用的 `mail()` 函数，因此这可能是你的首选。但一旦你意识到需要根据 RFC822 标准验证所有电子邮件地址，并且设计团队想要在邮件中发送图片附件时，你可能就开始重新权衡选择了。碰巧的是，`pear/mail` 支持这两个功能（后一个功能需结合 `mail_mime` 使用）。

许多程序员，包括我在内，常常过分强调创作原创代码，有时甚至损害了项目利益。

**注意**

不愿使用第三方工具和解决方案的态度，常常是机构层面固有的。这种对外部产品持怀疑倾向的现象，有时被称为“非我发明”综合征。

这种对原创性的强调，可能是导致可重用代码似乎被创造得多、实际用得少的原因之一。

高效的程序员将原创代码视为帮助他们成功完成项目的工具之一。这样的程序员会审视手头的资源，并有效利用它们。如果某个包能分担压力，那就是胜利。借用并改写 Perl 世界的一句格言：好程序员都是“懒人”。

## 和睦共处

萨特那句著名的格言“他人即地狱”，其真理性在某些软件项目中每天都在被印证。这或许描述了客户与开发者之间的关系，其症状表现为沟通不畅导致的种种问题，例如需求蔓延和优先级错位。但这句话也同样适用于那些即使沟通愉快、合作积极的团队成员，尤其是在共享代码的时候。

一旦项目有超过一个开发者，版本控制就成了一个问题。单个程序员可以直接在原有代码上工作，在开发的关键节点保存一份工作目录的副本。但只要引入另一个程序员，这种策略几分钟内就会失效。如果新开发者在同一个开发目录下工作，那么除非两人都格外小心，始终在不同的文件上工作，否则其中一个程序员保存时很可能会覆盖同事的工作成果。

或者，我们的两位开发人员可以各自获取一份代码库的副本，进行独立开发。这方法在需要整合两个版本之前一直没问题。除非开发者们处理的是完全不同的文件集，否则合并两条或多条开发分支的任务很快就会变成巨大的难题。

这正是 Git、Subversion 及类似工具的用武之地。使用版本控制系统，你可以签出自己那份代码库的副本，并进行开发，直到你对结果满意为止。然后，你可以用同事们所做的任何更改来更新你的版本。版本控制软件会自动将这些更改合并到你的文件中，并通知你任何它无法处理的冲突。一旦你测试了这个新的混合版本，就可以将其保存到中央仓库，使其可供其他开发者使用。

版本控制系统还能为你带来其他好处。它们会保存项目所有阶段的完整记录，因此你可以回滚到项目生命周期中的任何一点，或抓取其快照。你还可以创建分支，这样你就可以在维护稳定发布版本的同时，维护一个前沿的开发版本。

一旦你在项目中使用过版本控制，你将无法想象没有它如何进行另一个项目。同时处理项目的多个分支可能是一个概念上的挑战，尤其是在刚开始的时候，但其好处很快会变得显而易见。版本控制实在太有用了，不可或缺。我将在第 17 章中介绍 Git。

## 为你的代码插上翅膀

你是否曾因为代码构建过于困难而将其搁置？对于在原有环境中开发的项目尤其如此。这类项目深深嵌入其环境，密码、目录、数据库和辅助程序调用都直接编写在代码里。部署此类项目可能是一项艰巨的任务，需要程序员团队仔细检查源代码来修改设置，以便适应新环境。

通过提供一个集中式的配置文件或类，可以在一定程度上缓解这个问题，这样设置只需在一个地方更改即可。但即便如此，构建仍然可能是一件麻烦事。安装的难易程度将严重影响你所分发的任何应用程序的普及度。它也将阻碍或促进开发期间进行多次和频繁的部署。

就像任何重复且耗时的任务一样，构建应该被自动化。构建工具可以确定安装位置的默认值、检查和更改权限、创建数据库以及初始化变量等。事实上，构建工具几乎可以完成你需要将应用程序从分发源目录到完全部署的所有工作。

当然，这并不能免除用户将关于其环境的信息添加到代码中的责任，但它可以使这个过程变得如同回答几个问题或提供几个命令行开关一样简单。

像亚马逊 EC2 这样的云产品已经使得按需创建测试和模拟环境成为可能。为了充分利用这些资源，良好的构建和安装解决方案至关重要。如果你无法即时部署你的系统，那么能够自动化配置服务器也是毫无意义的。

开发者可以使用多种构建工具。PEAR 和 Composer 都管理安装（PEAR 是集中式的，Composer 安装到本地的 `vendor` 目录）。你可以为这两个系统创建自己的包，用户随后可以轻松下载并安装这些包。然而，构建远不止是将文件 A 放到位置 B 这个过程。

在第 19 章中，我将介绍一个名为 Phing 的应用程序。这个开源项目是流行的 Ant 构建工具的一个移植版，后者是用 Java 编写的，并用于 Java 项目。Phing 是用 PHP 编写的，适用于 PHP，但它在架构上与 Ant 类似，并使用相同的 XML 格式作为其构建文件。

Composer 能够极其出色地执行数量有限的任务，并提供尽可能简单的配置。Phing 起初看起来更令人生畏，但换来的是巨大的灵活性。你不仅可以使用 Phing 自动化从文件复制到 XSLT 转换的任何事情，而且如果你需要扩展该工具，你还可以轻松编写并整合自己的任务。Phing 使用 PHP 的面向对象特性编写，其设计强调模块化和易于扩展性。

构建工具与专为包管理或依赖管理设计的工具并非互斥。通常，构建工具在开发期间用于运行测试、执行项目维护工作，以及准备最终通过 PEAR、Composer 甚至基于发行版的包管理系统（如 RPM 和 Apt）部署的包。

### 标准

我之前提到，本书在本版本中将重点从 PEAR 转向了 Composer。这是因为 Composer 比 PEAR 好得多吗？我确实很喜欢 Composer 的很多方面，这些本身可能就足以左右这个决定。不过，本书转变的主要原因是：其他所有人都在转变。Composer 已成为依赖管理的标准。这一点至关重要，因为这意味着当我在 Packagist 上找到一个包时，我很有可能也能找到它的所有依赖项和相关包。我甚至能在那儿找到许多 PEAR 包。

因此，为依赖管理选择一个标准可以确保可用性和互操作性。但标准的适用性超越了包和依赖本身，还涉及系统的工作方式以及我们的编码方式。如果我们就协议达成一致，那么我们的系统和团队就能无缝地相互集成。而且，随着越来越多的组件在越来越多的系统中组合，这一点变得越来越重要。

当需要某种确定的日志处理方式时，采用最佳协议显然是理想的选择。但推荐标准的质量（它将规定格式、日志级别等）可能不如我们大家共同遵守它这一事实重要。如果你是唯一遵循最佳标准的人，那实施它也没什么好处。

在第 15 章中，我将更详细地讨论标准，特别参考由 PHP-FIG 小组管理的一组建议。这些 PSR（PHP 标准建议）涵盖了从缓存到安全的方方面面。在该章中，我将重点介绍 PSR-1 和 PSR-2，这是针对棘手的编码风格问题（你习惯把大括号放在哪里？对于别人要求你改变习惯，你有何感受？）提出的建议。然后，我将继续介绍绝对有益的 PSR-4，它涵盖了自动加载（对 PSR-4 的支持是 Composer 表现出色的另一个领域）。

## Vagrant

你的团队使用什么操作系统？当然，有些组织会强制规定特定的硬件和软件组合。但通常情况下，会是混合使用的。一位开发者可能使用运行 Fedora 的开发机器。另一位可能对它的 MacBook 赞不绝口，而第三位可能坚持使用他的 Windows 电脑（他大概喜欢用它来玩游戏）。

很有可能，生产系统会运行在完全不同的系统上——也许是 CentOS。

让一个系统跨多个平台工作可能很痛苦，而且如果这些平台没有一个与生产系统相似，这可能会带来风险。你肯定不想在上线后发现与生产操作系统相关的问题。当然，在实践中，你可能会先部署到预发布环境。即便如此，及早发现这些问题不是更好吗？

`Vagrant` 是一种利用虚拟化技术，为所有团队成员提供一个尽可能接近生产环境的开发环境的技术。启动并运行应该简单到只需调用一两个命令，最棒的是，每个人都可以继续使用他们最喜欢的机器和发行版（顺便说一句，我是 Fedora 爱好者）。

我将在第 20 章中介绍 `Vagrant`。

### 测试

当你创建一个类时，你很可能相当确定它能正常工作。毕竟，你会在开发过程中对它进行彻底测试。你还会让系统在包含该组件的情况下运行，检查它是否集成良好，以及你的新功能是否可用并按预期执行。

但你能确定你的类会像预期的那样持续工作吗？这看起来可能是个愚蠢的问题。毕竟，你曾经检查过你的代码一次；它为什么应该随意停止工作呢？嗯，当然它不会；没有什么事情是随意发生的，而且如果你从未向系统添加另一行代码，你或许可以松一口气。另一方面，如果你的项目是活跃的，那么你的组件的上下文不可避免地会发生变化，并且该组件本身极有可能以多种方式被修改。

让我们依次看看这些问题。首先，改变组件的上下文怎么会引入错误？即使在组件之间很好地解耦的系统中，它们仍然是相互依赖的。你的类所使用的对象会返回值、执行操作和接收数据。如果这些行为中的任何一个发生改变，对你类运行的影响可能会导致那种容易捕获的错误——那种系统崩溃并伴随一条包含文件名和行号的便捷错误信息的错误。然而，更隐蔽的是那种不会引起引擎级别错误，但确实会使你的组件混淆的改变。如果你的类基于另一个类的数据做出了一个假设，那么那个数据的改变可能导致它做出错误的决定。你的类现在就出错了，并且没有改变一行代码。

而且，你很可能会继续修改你刚刚完成的类。通常，这些修改是微小且显而易见的——实际上如此微小，以至于你觉得没有必要重新执行在开发过程中进行的仔细检查。无论如何，你可能已经忘记了它们，除非你以某种方式保留了它们（也许像我有时做的那样，在类文件的底部注释掉它们）。然而，微小的改变往往会造成意想不到的重大后果——如果你曾考虑过设置一个测试工具，这些后果也许就能被捕捉到。

测试工具是一组自动化的测试，可以应用于整个系统或其单个类。如果部署得当，测试工具有助于防止错误的发生和再次发生。一个单一的改变可能导致一连串的错误，而测试工具可以帮助你定位并消除这些错误。这意味着你可以有信心地进行修改，而不会破坏任何东西。当你对系统进行改进，然后看到一列失败的测试时，会感到相当满足。这些都是可能在你的系统中传播的错误，但现在它不必承受这些错误了。

## 持续集成

你有没有制定过一个让一切都变得美好的计划？你从一个任务开始：可能是编码，或者一个学校项目。它庞大而可怕，失败潜伏其中。但你拿出一张纸，把它分割成可管理的任务。你确定要阅读的书籍和要编写的组件。也许你用不同的颜色高亮这些任务。结果发现，单独来看，没有一个任务真正那么可怕。并且，随着你逐步规划，你逐渐征服了截止日期。只要你每天做一点，你就会没事。你可以放松了。

但有时，这个计划会具有护身符般的力量。你把它像盾牌一样举起来，保护自己免受怀疑，以及免受那种也许这次你会惨败的逐渐蔓延的恐惧。只有在几周后，你才意识到计划本身并没有魔力。你还得实际去做工作。当然，到那时，被计划令人安心的力量所麻痹，你已经让事情拖延了。除了制定一个新计划，别无他法。这次它不会那么令人安心了。

测试和构建也是如此。你必须运行你的测试。你必须构建你的项目，并定期在新的环境中构建它们；否则，魔力就不会起作用。

而且，如果编写测试是一种痛苦，那么运行它们可能是一件苦差事，尤其是当它们变得复杂且失败打断了你的计划时。当然，如果你更频繁地运行它们，你可能会有更少的失败，并且你遇到的那些失败很可能与你脑海中记忆犹新的新代码有关。

在沙盒环境中很容易变得安逸。毕竟，你所有的玩具都在那里：让你生活更轻松的小脚本、开发工具和有用的库。问题是，你的项目也可能在你的沙盒中变得过于安逸。它可能开始依赖未提交的代码，或者你从构建文件中遗漏的依赖项。这意味着除了你工作的地方，它在任何其他地方都是损坏的。

唯一的答案是构建、构建、再构建。并且每次都要在一个相对干净的环境中进行。

当然，建议这样做是一回事，实际去做又是另一回事。程序员这个群体通常喜欢编码。他们希望将会议和日常维护工作降到最低。这就是持续集成 (CI) 发挥作用的地方。 CI 既是一种实践，也是一套尽可能让实践变得简单的工具。理想情况下，构建和测试应该完全自动化，或者至少可以通过一条命令或一次点击来启动。任何问题都会被跟踪，并且在问题变得过于严重之前你会收到通知。我将在第 21 章中更多地讨论 CI。

## 总结

开发者的目标始终是交付一个可运行的系统。编写好的代码是实现这个目标的重要部分，但这并非全部。

在本章中，我介绍了如何使用`Composer`和`Packagist`进行依赖管理。我还讨论了两个重要的协作辅助工具：`Vagrant`和版本控制。我阐述了为什么版本控制需要自动化构建，并介绍了`Phing`，它是Java构建工具`Ant`的PHP实现。最后，我讨论了软件测试并介绍了持续集成，这是一套用于自动化构建和测试的工具。
