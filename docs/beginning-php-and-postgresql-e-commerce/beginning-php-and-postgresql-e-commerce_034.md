# 排版后内容

在此场景中，部分逻辑从业务层转移到了数据层。在很多其他情况下，你可以选择将某些逻辑放在某一层、另一层，或同时放在两层。大多数情况下，实现三层架构并没有唯一的最佳方式，你需要根据个人偏好或外部限制做出折衷或选择。

此外，有时即便你知道实现某项功能的“正确”方式（从架构角度看），你也可能会选择打破规则以提升性能。通常来说，如果这种方式能提升性能，那么*稍微*打破各层之间的严格界限是可以接受的（例如，将部分业务规则添加到数据层，或者反过来），*前提是*这些规则不太可能随时间改变。否则，将所有业务规则保留在中间层是更好的选择，因为它能生成更“干净”、更易于维护的应用程序。

最后，不要试图从表示层直接访问数据层。这是一个常见错误，也是导致系统复杂、难以维护且缺乏灵活性的最快途径。在许多数据访问教程或入门材料中，你会看到如何通过简单的用户界面应用程序执行基本数据库操作。在这类程序中，所有逻辑可能都写在一个简短的单文件中，而非分散到不同层。

尽管这些材料可能很好，但请记住，这些文本大多旨在教你如何完成不同的独立任务（例如访问数据库），而不是如何正确创建灵活且可扩展的应用程序。

## HatShop 的三层架构

为 HatShop 网站实现三层架构将有助于实现本章开头列出的目标。乍看之下似乎僵化的系统所强加的编码规范，从长远来看能带来极佳的灵活性和可扩展性。

将应用程序的主要部分拆分为独立的较小组件，有助于提升可重用性。在为网站添加新功能时，你通常会多次发现可以重用某些现有部分。无需大幅改动现有代码即可添加新功能，这本身就是可重用性的一个良好范例。

三层架构的另一个优势是，如果实施得当，整个系统对于变化具有很好的鲁棒性。当某一层中的组件发生变化时，其他层通常不受影响，甚至在极端情况下也是如此。例如，如果出于某种原因更换了后端数据库系统（比如经理决定使用 Oracle 替代 PostgreSQL），你只需更新数据层，或许再稍微调整一下业务层即可。

## 为什么不使用更多层？

我们迄今为止所讨论的三层架构是 *n* 层架构的一个特定（也是最流行的）版本。**\*n\*-层架构**指的是将解决方案拆分为数量为 *n* 的逻辑层。在复杂项目中，有时将业务层拆分为多个层是合理的，从而产生超过三层的架构。

然而，对于我们的网站而言，坚持使用三层设计是最合理的，它既能提供大部分优势，又无需花费过多设计时间或使用复杂的框架代码层级来支撑架构。

或许采用更复杂、更精细的架构，你能为应用程序实现更高水平的灵活性和可扩展性，但在开始实施任何内容之前，你需要花费更多时间进行设计。与任何编程项目一样，你必须在设计架构所需的时间与实施架构所需的时间之间找到恰当的平衡。三层架构最适合中等复杂度的项目，例如 HatShop 网站。

你可能还会问相反的问题：“为什么不使用更少的层？”两层架构，也称为*客户端-服务器*架构，适用于复杂度较低的项目。简而言之，两层架构需要更少的规划时间，并在初期允许更快的开发，但从长远来看，它生成的应用程序更难维护和扩展。由于我们预计将来需要扩展应用程序，因此客户端-服务器架构不适合我们的应用程序，本书将不再进一步讨论。

既然大致架构已经明确，我们来看看你将使用哪些技术和工具来实现它。在简要讨论技术之后，你将通过创建网站的第一个页面和后端数据库，来搭建表示层和数据层的基础。在第 3 章开始创建网站的产品目录时，你将在三个层中分别开始实现一些实际功能。

### 使用 PHP 生成动态 Web 内容

PHP 是一种用于构建动态、交互式 Web 内容的开源技术。其在官方 PHP 网站（`http://www.php.net`）上的简短描述是：“PHP 是一种广泛使用的通用脚本语言，特别适用于 Web 开发，并可嵌入到 HTML 中。”

PHP 代表 *PHP: Hypertext Preprocessor*（是的，这是一个递归缩写），可在其官方网站免费下载。我们在附录 A 中包含了 PHP 的完整安装说明。由于我们使用 PHP 构建动态网站，你还可以在附录 A 中学习如何安装 Apache 以及如何将 PHP 与之集成。

PHP 的故事始于 1994 年，是一段成功的经历。促成其成功的因素包括：

- PHP 是免费的；特别是与 Linux 服务器软件结合使用时，PHP 可被证明是一种构建动态 Web 内容的高性价比技术。
- PHP 的学习曲线比其他脚本语言更短。
