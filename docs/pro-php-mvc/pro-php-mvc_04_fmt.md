## 第 1 章：MVC 简介

软件开发并不是一个新概念。据说，埃达·洛夫莱斯在 19 世纪中期为分析机（查尔斯·巴贝奇设计的第一台机械计算机原型）编写了第一个计算机程序。

从那时起已经过去了很长时间，软件开发已发展成为可以说对人类发展贡献最大的领域之一。

设计优秀的软件很难。它需要考虑到你需要构建的应用程序的方方面面，并且由于需要足够具体以满足当前需求从而完成工作，同时又要足够通用以应对未来的问题，这使得事情变得更加复杂。许多经验丰富的开发者都遇到过这些问题，随着时间的推移，一些通用的模式已经浮现出来，有助于解决这些问题。

结构建筑师克里斯托弗·亚历山大首次以可以应用于软件开发的方式描述了模式。他说：“每个模式都描述了我们环境中反复出现的一个问题，然后描述了该问题解决方案的核心，这样你就可以无数次地使用这个解决方案，而不会两次都以完全相同的方式去做。”他可能是在谈论房屋或城市，但他的话抓住了我们在考虑如何为 Web 应用程序构建一个稳固、安全且可复用的框架时意图的精髓。

### 什么是 MVC？

MVC（模型-视图-控制器）是一种软件设计模式，它围绕三种主要组件类型在编程语言（如 PHP）中的相互连接而构建，通常重点关注面向对象编程（OOP）的软件范式。这三种组件类型被松散地称为模型、视图和控制器。让我们分别讨论它们，然后看看它们是如何组合在一起的。

模型是存放应用程序所有业务逻辑的地方。业务逻辑可以是任何特定于应用程序如何存储数据，或使用第三方服务以满足其业务需求的内容。如果应用程序需要访问数据库中的信息，执行该操作的代码将存放在模型中。例如，如果需要获取股票数据或发布关于新产品的推文，这些代码也将存放在模型中。

视图是存放应用程序所有用户界面元素的地方。这可以包括我们的 HTML 标记、CSS 样式表和 JavaScript 文件。用户看到或与之交互的任何内容都可以存放在视图中，有时用户看到的内容实际上是同一个请求中许多不同视图的组合。

### 控制器

控制器是连接模型和视图的组件。控制器将模型的业务逻辑与视图的用户界面元素隔离开来，并处理应用程序如何响应用户在视图中的交互。控制器是这三个组件中的入口点，因为请求首先被传递给控制器，然后控制器会实例化完成应用程序请求所需的模型和视图。参见图 1-1。

- 我们的社交网络应该易于使用，并对其用户有吸引力。因为我们是为 Web 构建它，所以我们将使用标准网站技术，例如用于标记的 HTML、外部链接的 CSS 样式表和用于行为的外部链接的 JavaScript 文件。所有这些元素都将出现在视图中。

- 我们应用程序的模型和视图必须在不相互干扰的情况下连接在一起。此外，应用程序需要一种方式来响应用户在视图中的交互，并将相关数据持久化到模型中。控制器用于此目的。

希望这能以一种合理的方式说明这个概念。在本书中，我们将更详细地研究每个部分。当我们解构使框架运行所需的代码时，这个简单的社交网络也将作为一个一致的例子。

### MVC 的优势

在不了解为什么要使用 MVC 的情况下，解释 MVC 是什么是没有意义的。还记得我之前提到的 Christopher Alexander 的模式吗？MVC 是本书中将解释的众多模式之一，但要理解这种设计模式的用处，我们必须着眼于它有助于缓解的问题。

如果你想到一支运动队，你可能会意识到它本质上是一大群球员，他们各自扮演好自己的角色以推动球队前进。优秀的运动队需要每位球员尽最大努力发挥自己的作用，从而推动整个团队前进。

Web 是一个开放的竞技场。它允许大大小小的企业相互竞争，而规模并不代表工作质量。这意味着许多拥有小型开发者团队的小公司有机会构建大型 Web 应用程序。这也意味着许多大公司可以同时让许多人参与大型 Web 应用程序的开发。在所有这种多任务处理和/或团队协作中，应用程序的各个方面（本应分离）常常会相互干扰，并且需要比严格必要更多的时间和精力来推进。

任何复杂的 Web 应用程序都有许多方面。有设计，它激发用户对产品的兴趣。有执行业务操作（例如处理销售商品和给购物者开具发票）所需的业务逻辑。然后是应用程序的持续改进、更新、修复错误和总体优化过程。

在任何非结构化的应用程序中，这些领域往往会在一种不连贯的混乱中融合在一起。当需要更改数据库以适应新产品线，或者公司决定重塑品牌时，这不仅会影响它应该影响的代码。更多的开发者必须介入，以确保应用程序一部分的更改不会立即破坏应用程序的其他部分。本应只影响一小段代码的更改最终会蔓延到各种奇怪且有问题的领域。

这就是 MVC 试图解决的问题。它为应用程序的所有代码和功能定义了严格的容器。当对数据库代码的更改被隔离在模型中时，视图和控制器不会中断。当应用程序的设计发生巨大变化时，其控制器和模型将免受破坏性更改的影响。

■ **注意** 一个基于 MVC 的优秀应用程序要成功，需要的不仅仅是一个好的 MVC 框架。它需要开发者愿意遵守规则，仔细思考他们的代码应该放在哪里，而不是仅仅将其扔进代码库。我们只能设计结构，就像建筑师设计一座漂亮的房子一样。最终要靠使用我们框架的开发者来保持秩序。

既然我们更多地了解了为什么要使用 MVC，那么让我们看看编写自己框架的一些流行替代方案。

### 流行的 MVC 框架

有许多优秀的 PHP 框架可用，但如果我们将视角限制在三个框架上，我认为我们可以很好地了解它们的共同之处以及各自的独特之处。这些并不是最好或唯一的 PHP MVC 框架，而仅仅是 PHP MVC 开发不同方法的一个良好横截面。

#### CodeIgniter

CodeIgniter 是我们将研究的第一个也是最简单的框架。它由 EllisLab 开发和维护，可以描述为一个开源（尽管受到严格控制）框架，构成了 EllisLab 高级 CMS（内容管理系统）ExpressionEngine 的基础。

它已经存在了很长时间，但自从我第一次使用它以来，其理念多年来几乎没有改变。它力求保持小巧的体积、出色的开发者文档和高质量的代码。它不像我们将要讨论的其他一些框架那样受欢迎，部分原因是 EllisLab 管理 CodeIgniter 社区的方式。他们最近开始通过举办新的会议和增加员工来解决这个问题，这个框架的前景正在好转。

它还启发了其他框架，孕育了诸如 KohanaPHP 之类的项目。

> **注意** 你可以从 [`codeigniter.com`](http://codeigniter.com) 下载 CodeIgniter。你还可以在 [`ellislab.com`](http://ellislab.com) 了解更多关于 EllisLab 和 ExpressionEngine 的信息。最后，你可以在 [`kohanaframework.org`](http://kohanaframework.org) 了解更多关于 KohanaPHP 的信息。

## Zend Framework

Zend Framework 是一个由松散耦合的代码库组成的广泛集合，可以构成 MVC 架构的基础。相对于其他流行的 MVC 框架，理解和掌握 Zend Framework 需要付出相当大的努力。它由 Zend Technologies 开发，并享有庞大、稳定的社区和广泛采用的所有优势。

像 `CodeIgniter` 这类框架力求轻量，仅保留核心功能，而 `Zend Framework` 则包含了帮助开发者使用多种第三方服务和 API 的库。

> **注意** 你可以在 [`framework.zend.com`](http://framework.zend.com) 下载 `Zend Framework`。你也可以在 [`zend.com`](http://zend.com) 了解 `Zend` 的更多信息。

## CakePHP

`CakePHP` 可以说是这三个框架中最受欢迎的。与前两个框架不同，它不受任何单一企业实体的管辖。它拥有庞大的社区，并被广泛采用。

它更倾向于约定优于配置，这意味着许多细节会被预设并自动化。

这一点在框架的各个方面都显而易见，你常常会惊讶于 `CakePHP` 自动执行了你并未要求的事情，这其中既有好处也有坏处。这意味着你可以快速开发应用程序，但当你需要进行非常具体的修改时，也可能会遇到困难。

这一点甚至体现在其代码生成命令行工具 `Bake` 上。只需遵循命令行提示，并使用默认参数和行为填空，它就能在几分钟内生成一个可运行的应用程序。

> **注意** 你可以在 [`cakephp.org`](http://cakephp.org) 下载 `CakePHP`。

## 设计模式

我们将重点探讨 MVC 设计模式。为了实现它，我们需要在构建框架的库中使用其他更简单的设计模式。我们即将回顾的设计模式通常可以应用于过程式开发，但我们将从面向对象编程的角度来审视它们。

这意味着我们将处理类（包含属性和执行功能的蓝图）以及它们之间的相互交互。如果你对后续某些概念不熟悉，可能需要查阅语言入门指南或参考网站。

> **注意** 如果你想了解更多关于面向对象编程或后续的其他关键词/概念，可以在 [`en.wikipedia.org/wiki/Object-oriented_programming`](http://en.wikipedia.org/wiki/Object-oriented_programming) 上阅读更多信息。

### 单例模式

在构建面向对象软件时，我们会处理许多类。虽然理想情况下，设计这些类时应允许多个实例同时激活，但有时，出于特定目的或上下文，我们实际上只需要一个类的实例。

`单例模式` 是一种确保一个类在同一时间只能有一个实例的设计模式。传统的 `单例` 类在内部静态属性中维护自身的唯一实例，并且不能像非单例类那样以常规方式进行实例化（或克隆）。`单例` 类有一个特殊的实例访问方法，该方法返回内部实例属性，或者创建一个新实例来返回和存储。参见图 1-2。

### 注册表模式

`注册表` 是一个可以存储并返回标准类实例的类。可以把它想象成一个团队经理，他可以将球员从场上替换下来，并根据需要派上新球员。我们使用 `注册表` 类来管理有限数量的类实例，这样我们就不需要重复实例化 `注册表` 中已包含实例的类。

`注册表` 类的另一种理解方式是，它帮助我们将普通类当作 `单例` 来对待，而无需将这些普通类变成 `单例`。我们可能会遇到需要两个类实例的情况。例如，我们需要连接两个独立的数据库，但又不想重复建立连接，这时就可以使用 `注册表`。参见图 1-3。

### 工厂模式

`工厂` 是一个提供单一接口用于创建任意数量实例的类，而无需实际指定实例应属于哪个类。`工厂` 会根据输入或内部逻辑来选择要实例化的类。

### 观察者模式

`观察者` 模式描述了一种由发送者和接收者组成的结构。当发送者的状态发生变化时，它会向与之关联的接收者发送消息，通常通过调用接收者某个函数来实现。

## 创建我们自己的框架

你可能会想，既然市面上已经有这么多优秀的选择，我们为什么还需要创建自己的框架。其原因在于，这样我们才能理解 MVC 的基本原理。

随着我们对这些原理的深入了解，我们就能更好地理解现有的优秀 MVC 框架为何采用当前的做法。我们学习的目标不是在 `Zend Framework` 或 `CakePHP` 中创建应用程序，而是学习 MVC 的工作原理，并由此理解这些框架是如何在我们期望的 MVC 框架构建方式基础上进行构建（或偏离）的。

我们的目标不是为可投入生产使用的框架列表再添一员。我们的目标是学习这些框架“做了什么”以及“为什么这么做”，以便在为我们下一个大项目选择框架时，能够做出明智的决策。

> **注意** 欢迎你将我们构建的框架用于生产环境，但我建议你谨慎行事，除非你准备投入大量时间来满足框架的安全性和稳定性要求，因为那些流行的框架也是花费了多年时间才达到的。

### 目标
