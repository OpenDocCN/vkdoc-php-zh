# 测试与胖模型

编写更好代码的一个概念是拥有胖模型和薄控制器。

这意味着控制器应该非常薄，仅作为请求和响应之间的转换层。因此，业务逻辑放在模型中，由控制器频繁使用。

另一个概念是越小越好。这意味着我们应该有短小的方法。有人主张每个方法最多三行，另一些人则推荐最多十行。短方法更易于阅读、理解、维护和测试。长方法容易违反单一职责原则。

所有这些意味着我们将拥有包含许多短小方法的胖模型，每个方法只负责一件事。而且你可能会为所有模型方法编写测试。

测试将帮助你识别自己遵循的导致问题的糟糕编码模式。

CakePHP 遵循模型-视图-控制器（MVC）模式，即使编程模式对你来说很陌生，它也提供了良好的支持并帮助你遵循。假设我们想要一个按用户名首字母分组的用户列表。例如，A: Amindala，B: Ben Kenobi，Boba Fett，等等。也许我们想要一个链接列表，以便用户点击某人的名字时，会被重定向到该用户的个人资料页面。

为满足此需求，我们应该查询用户表以获取用户名，然后将结果提供给视图，并构建 HTML 列表。

与数据库的通信应在模型中进行。所有`find()`调用都应放在模型中。

控制器将检查用户权限，并设置变量供视图显示或闪现错误消息。

视图应实际构建 HTML 列表和链接。视图不应执行任何业务逻辑，只应输出内容。如果在控制器或模型中使用了`print`（或`echo`）语句，就破坏了 MVC 模式。同时，大多数视图代码应仅输出内容，不做其他事情。计算表格汇总或平均值是例外情况。
