# 第 8 章 ■ 模板

■ **注意** 实现模板解析器的方法有很多，我们选择了一种能够很好地满足应用程序需求的方式。仅仅这个主题就足以写成一本书，因此重要的是要认识到，我们的方法是在现有约束条件下功能最丰富的。

### 问题

1. 在本章中，我们为`实现`和`解析器`分别创建了独立的类。为什么这比使用单个类更好？

2. 是否有办法修改模板解析器，使其能够处理不带 PHP `$` 前缀的变量？

3. 是否可能在我们设计的模板方言中实现其他控制流结构，例如 `switch` 语句？

### 答案

1. 这是一种很好的做法，因为它为模板方言和解析器未来的扩展性留出了空间。我们可以轻松地替换成不同的方言类，而解析器依然能正常工作。实际上，我们将在后续章节中扩展这个方言，所以这非常合适！

2. 模板解析器在后台所做的全部工作就是字符串替换。理论上，我们可以移除这些 PHP 的痕迹，但这需要对传入控制结构的参数进行智能解析。我们不能再像现在这样，简单地将这些参数列表直接放入解析后的模板中。

3. 这是可行的。当向这个模板方言和解析器添加新的控制流结构时，你只需要思考模板标签如何转化为可执行的 PHP 代码。这正是当前标签所做的事情。

### 练习

1. 模板方言包含一套丰富的标签，用于控制模板中的执行流程。尝试用 `switch` 标签进行扩展。

2. 使用本章创建的类，现在已经可以解析包含变量数据的字符串模板了。试试看吧！
