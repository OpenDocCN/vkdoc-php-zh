# 6. 解耦 Drupal 的缺点

鉴于目前围绕解耦 Drupal 的所有热潮和好评，很容易在没有深入评估的情况下就回避解耦 Drupal 的陷阱。如果你需要 Drupal 的前端功能或其他依赖于 Drupal 默认前端存在的关键功能，那么选择仅将 Drupal 用于其 Web 服务能力和内容仓库，可能会危及你的整个架构。其他风险也同样存在。

## 额外的故障点

通常，单体式 Drupal 实现部署在当今网络上无处不在的 LAMP（Linux, Apache, MySQL, PHP）技术栈上。另一方面，在完全解耦架构中作为客户端的 JavaScript 消费者应用程序，则必须使用像 MERN（MongoDB, Express, React, Node.js）或 MEAN（Angular 替代 React）这样的 Node.js 技术栈。对于有不同需求的本地移动应用或物联网应用，可能需要完全不同的其他解决方案。

因此，在你的组织基础设施中引入一个额外的托管技术栈，不仅对那些资源有限的人来说很困难；它还会在你的架构中引入一个额外的故障点。例如，如果充当 Web 服务提供者的 Drupal 站点在没有适当缓存的情况下发生故障，那么你的应用程序向用户提供的数据可能会过时或无法访问。同样，作为 Drupal 内容唯一通道的消费者应用程序如果遇到停机，除非你的 Drupal 站点也公开可访问，否则将导致用户无法访问你的内容。

## 安全与输入清理

虽然绝大多数现代 JavaScript 和应用框架都包含了抵御跨站脚本攻击的方法，包括输入清理等措施，但无论是完全解耦还是渐进式解耦的 Drupal，都需要你仔细审视架构的安全性影响。例如，尽管 Drupal 为所有表单字段提供了开箱即用的表单验证和文本清理功能，但这仅在单体架构中可用。

如果你或你的组织选择使用定制框架——或者在没有框架帮助的情况下使用“原生”JavaScript——那么用户生成输入的安全性就会成为一个潜在的巨大风险，这是一个需要在此着重强调的危害。与其让 Drupal 来承担繁重的工作，你的自有方案需要进行充分的研究，以评估你是否已采取令人满意的步骤来确保用户、消费者应用程序，以及最终整个架构的安全。

## 情境化编辑与管理

Drupal 8 一些最引人注目的功能包括原地编辑（称为 Quick Edit 模块）以及伴随某些页面组件的配置菜单（称为 Contextual Links），这些是为 Drupal 管理提供情境化工具的模块中的一部分。在单体架构中进行内容预览时，这些界面允许站点构建者和内容编辑者在查看实时结果的同时调整内容，或者从便利的可视化预览中访问管理页面。

在完全解耦的 Drupal 架构中，这些情境化工具将不再可用，除非在消费者应用程序上重建它们，这会导致功能重复。因此，为 Web 体验使用与 Drupal 不同的前端，会将部署此类界面的责任转移给前端开发人员，他们可以提供这些工具的替代品或复制品，或者在编辑者的体验中注明其不可用。渐进式解耦方法在一定程度上缓解了这个问题，尽管它存在一个“黑箱”问题，即页面解耦区域中包含的内容无法按照 Drupal 管理员通常的预期进行编辑或管理。

## 布局与显示管理

Drupal 8 的特性包括提供布局和显示管理功能的核心模块与贡献模块，这些模块提供了一系列选项，用于提供可变内容显示（`Display Suite`）或构建由内容窗格组成的布局（`Panels`）。由于这些模块需要对 Drupal 的标记进行大量控制，因此它们需要与 Drupal 的表示层紧密耦合。

将诸如 `Panels` 和 `Display Suite` 之类的模块从编辑流程中移除，意味着布局管理变成了开发人员而非编辑者需要考虑的问题。这对于那些无法接触到能够协助实施布局变更的开发人员的营销团队来说，会造成相当大的挑战。为了让编辑者能够继续操作布局而不仅仅是内容，你的组织要么需要使用 Drupal 的表示层，要么在消费者应用程序中重建布局管理功能，要么通过像 `RESTful Panels` 这样的模块，以可摄入数据的形式暴露布局配置。

## 可预览的内容工作流

一个健壮的 CMS 最重要的基础之一，是能够创建并按照编辑内容工作流进行操作，通过该工作流，可以方便地预览内容状态（例如草稿和“审核中”），而不会过早地发布可能受保护的内容。

通过使用与 Drupal 不同的前端来抛弃 Drupal 的内容预览和内容工作流功能，对于一个希望在内容工作流中实现可预览功能的编辑团队来说，会带来相当大的挑战，尤其是在他们可能已经习惯于在传统 CMS 中工作多年之后。存在几种替代方案，例如提供一个额外的私有预发布环境以允许部署不同的内容，或者扩展 Web 服务以通过二级认证或独特的查询参数来暴露差异化内容。

## 系统通知

Drupal 另一个关键特性是其强大的通知系统，该系统会显示 Drupal 系统进程期间出现的任何问题的信息，尤其是需要立即关注的严重系统错误。虽然 Drupal 中存在一个用于获取 watchdog 日志的 REST 资源，但这些资源只能提供管理员应审查的有限数量的可能问题。此外，在完全解耦的 Drupal 环境中，Drupal 系统消息（通常在渲染页面的顶部高亮显示）是无法访问的。

为了保持对 Drupal 中可能出现的潜在问题（尤其是可能影响向消费者应用程序传输数据的问题）的畅通无阻的感知，仔细监控 Drupal 后端界面中的系统消息非常重要，因为如果没有大量自定义代码，这些消息将不可用。在渐进式解耦的设置中，提供这些消息则不那么令人担忧，因为 Drupal 确实处理了部分渲染，并且只需要一个可以显示此类系统消息的区域。

## 单体架构的性能优势

Drupal 8 最引人注目的特性之一是缓存标签（见第 25 章），也称为可缓存性元数据，它允许开发人员定义对 Drupal 管理数据的依赖关系，并允许使严重依赖其中包含的细粒度内容的项目失效。例如，BigPipe 贡献模块通过基于各个页面组件的差异化可缓存性来提供页面的渐进式加载，从而缩短了首次绘制的时间。

这样的能力赋予了 Drupal 在页面加载期间实现显著性能提升的手段。有时，BigPipe 可以通过实现与 JavaScript 应用程序中预期类似的页面加载性能，来缓解开发人员对单体式 Drupal 性能的担忧。这种依赖于可缓存性元数据的渐进式加载对于完全解耦实现的开发人员来说是不可用的，但渐进式解耦构建在某些情况下可以利用此特性。当然，对于那些没有现有 Drupal 站点或不熟悉 Drupal 功能集的人来说，这个劣势可能不太重要。

## 无障碍与用户体验

最后，但或许是最重要的一点，Drupal 在无障碍性和用户体验方面的努力，包含了对于标记语言及其如何呈现给残障人士和辅助技术用户的最大程度考量。例如，Drupal 使用 ARIA 角色和其他技术，确保了所有 Drupal 内容对于屏幕阅读器用户来说都是可用的。此外，Drupal 历史上对可用性的持续关注意味着，任何使用 Drupal 的 `Form API` 的人，都能受益于一套标准化且经过实战检验的最佳实践。

在完全解耦和渐进式解耦的环境中，当涉及 JavaScript 应用程序时，标记语言和用户体验需要更多的思考，因为 Drupal 不再提供现成的前端代码或一套可依赖的核心界面组件和交互方式。这导致前端开发者需要在没有 Drupal 帮助的情况下，打造合适的用户体验和强大的无障碍性。幸运的是，近年来 JavaScript 框架在无障碍标记方面取得了显著进步。

### 结论

尽管解耦 Drupal 可以为您的团队以及构建数字体验生态系统的目标带来广泛收益，但它也伴随着某些权衡取舍，特别是当您将 Drupal 用作独立的内容服务而没有对应的前端时。在完全解耦的情况下，放弃基于 Twig 的前端可以实现更好的关注点分离和流水线式开发，但这迫使开发者必须更加关注无障碍性和用户体验的问题。

完全解耦 Drupal 会引入大量问题，这些问题应让任何利益相关者深思，并要求进行坦诚的评估。例如，额外的主机堆栈引入了第二个故障点（尽管您添加的每个基础设施不同的消费者都是如此）。安全问题，例如对用户生成内容的净化处理，也值得仔细审视。对于编辑和管理员而言，更关键的是，Drupal 中依赖于其表示层的那些关键功能，例如上下文工具、布局和显示管理、可预览的内容工作流以及系统通知，除了在管理界面内，在其他地方都会消失，除非它们作为副本在消费者应用程序中重建。

渐进式解耦通过提供一些解决方案来缓解其中的部分问题；例如，上下文工具、布局和显示管理、可预览内容以及系统通知都得以保留，不过“黑盒”问题是编辑们需要关注的一个来源。

换句话说，如果您选择解耦 Drupal，无论是完全解耦还是渐进式解耦，请准备好与一个高度胜任的开发团队合作，该团队应同时精通 Drupal 和前端技术，并在您推进解耦 Drupal 架构的过程中，尝试使用自定义或贡献的解决方案。

在第二部分中，我们将通过深入研究 Drupal Web 服务背后的细节来解耦 Drupal。我们首先转向 Drupal 核心中的核心 REST 模块，这些模块包括支持符合 Drupal 数据模型和 RESTful 原则的、遵循 HAL 规范的 JSON 响应的能力。然后，我们将注意力转向主要的贡献模块，例如 `JSON API`、`GraphQL` 和 `RELAXed Web Services`，它们各自提供了与 Drupal 内容进行交互的不同能力，以及身份验证机制。
