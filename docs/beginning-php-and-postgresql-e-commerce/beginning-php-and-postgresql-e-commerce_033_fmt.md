# 三层架构的魅力

通常，架构指的是根据功能将应用的每个功能块拆分成独立的组件，并将每种组件分组到单个逻辑层中。

你将为你网站创建的几乎每个模块都将包含来自应用服务器的这三个层的组件：

- 表示层
- 业务层
- 数据层

**表示层**包含网站的用户界面元素，并包含管理访问者与客户业务之间交互的所有逻辑。这一层让整个网站感觉充满活力，其设计方式对网站的成功至关重要。因为你的应用是一个网站，所以它的表示层由动态网页组成。

**业务层**（也称为*中间层*）接收来自表示层的请求，并根据其包含的业务逻辑将结果返回给表示层。几乎表示层中发生的任何事件通常都会调用业务层。

除了可由表示层本地处理的事件（如简单的输入数据验证等）之外，其他调用均由表示层发出。例如，如果访问者正在执行产品搜索，表示层会调用业务层并告知：“请将与搜索条件匹配的产品发回给我。”几乎总是如此，业务层需要调用数据层来获取信息，才能响应表示层的请求。

**数据层**（有时也称为*数据库层*）负责管理应用程序的数据，并在收到请求时将其发送给业务层。对于 `HatShop` 电子商务网站，你需要存储关于产品（包括其类别和部门）、用户、购物车等数据。几乎每个客户端请求最终都会导致数据层被询问以获取信息（除非之前检索到的数据已在业务层或表示层缓存），因此拥有一个快速的数据库系统至关重要。在第 3 章和第 4 章中，你将学习如何设计数据库以达到最佳性能。

这些层纯粹是逻辑上的——对每一层的物理位置没有限制。理论上，你可以自由地将整个应用程序及其所有层放在一台服务器上，或者如果应用程序允许，也可以将每一层放在不同的机器上。第 17 章将解释如何使用 `XML Web Services` 集成其他网站的功能。`XML Web Services` 允许跨多个服务器轻松集成功能。

三层架构模型中一个重要的约束是，信息必须在各层之间按顺序流动。表示层只能访问业务层，绝不能直接访问数据层。业务层是中间的“大脑”，负责与其他层通信，并处理和协调所有信息流。如果表示层直接访问数据层，就破坏了三层架构编程的规则。在实现三层架构时，你必须保持一致并遵守其规则，才能获得益处。

## 一个简单示例

通过仔细分析一个简单的示例，可以更容易理解数据在各层之间如何传递和转换。为了使示例与我们的项目更相关，我们来分析一个在 `HatShop` 中实际会发生的情况。这个场景对于三层应用程序来说是典型的。

像大多数电子商务网站一样，`HatShop` 将有一个购物车，我们将在本书后面讨论它。目前，只需知道访问者会通过点击 `Add to Cart`（加入购物车）按钮将产品添加到购物车。图 2-2 显示了点击该按钮时信息如何在应用程序中流动。

## 数字的含义是什么？

有趣的是，每个层对同一信息片段的解读方式不同。对于数据层来说，其存储的数字和信息毫无意义，因为它只是一个保存、管理和检索数字、字符串或其他数据类型的引擎——而不是产品数量或产品名称。在前一个例子的上下文中，产品数量 `0` 对数据层来说只是一个简单的纯数字，没有任何意义（它仅仅是 `0`，一个 32 位整数）。

当业务层读取这些数据时，数据才具有意义。当业务层向数据层询问产品数量并得到 `0` 的结果时，业务层将其解读为“嘿，没有库存了！”。最后，这些数据由表示层封装成漂亮的视觉形式，例如显示一个标签，上面写着：“抱歉，目前该产品无法订购。”

## 为正确的层配置正确的逻辑

由于每一层都包含自己的逻辑，有时很难决定各层之间的界限究竟在哪里。在前面的场景中，与其在业务层读取产品数量并根据该数字决定产品是否可用（这会导致两次数据层调用，也就意味着两次数据库调用），不如使用一个名为 `add_product_if_available`（如果可用则添加产品）的单一数据层方法，该方法仅当产品有库存时才将其加入购物车。
