# 2. 什么是云

对于“上云”的含义，存在一些混淆。有些人甚至错误地认为，只要在互联网上就是在利用云技术。

云技术与其他类型互联网托管服务的本质区别在于，云服务至少提供了快速扩展应用的能力。过去，开发者通常会将他们的网络应用部署到在特定设施中购买或租赁的固定服务器上。虽然可以添加更多设备，但这始终需要相当多的时间和精力。开发者常常需要为服务器筹备大量资金、购买、配置并部署它们。这个过程可能需要数周甚至数月。即使直接从托管公司购买，从整理报价到让一切正常运行，也可能需要一周多的时间。

云计算的承诺在于，无需经历为新服务器进行物理安装的过程，即可在瞬间、或在几分钟/几小时内（而非几天、几周或几个月）获得额外的容量。对于某些解决方案，你甚至无需操心机器——云解决方案会自动根据你的需求，将你的应用扩展到所需数量的机器上。对于其他方案，只需点击几下，即可请求、创建镜像并启动一台新机器，这台机器是现有某台机器的副本，然后将其添加到你的网络应用中。

无论如何，云的核心思想是即时、自动化的可扩展性和灵活性。

许多云提供商甚至提供对多个数据中心的访问。你想要一组服务器在美国，另一组在欧洲？没问题。只需点击几下，就能完成设置。

## 2.1 基础设施即服务

云计算常常令人困惑，因为存在几种不同类型的云计算，每种都有其自身的优缺点。云服务类型之间的区别主要基于所提供的抽象层级。

本书将重点介绍的最基本的云服务类型被称为*基础设施即服务*，缩写为 IaaS。IaaS 意味着你只需点击一个按钮，就可以购买和部署基础设施组件（服务器、负载均衡器、防火墙等）。这是通过服务器虚拟化技术实现的。

服务器虚拟化允许 IaaS 服务公司部署一台非常大的服务器，并将其拆分成多个较小的服务器。每台服务器运行一个*虚拟机监视器*程序，这使公司能够快速、自动地拆分服务器。一家公司可能会拿一台拥有 16 个处理器和 64 GB 内存的计算机，将其拆分为 4 台虚拟机，每台拥有 4 个处理器和 16 GB 内存。

这样做有三个好处。第一是空间。通过购买最大的机器，IaaS 公司为每个机架单元提供了最大的计算能力，这在服务器机房中是宝贵的资源。因此，通过购买一台大机器并将其拆分为四台较小的机器，他们仅使用了原本可能需要的四分之一的空间。

第二是每 CPU 核心的成本。通过将如此多的 CPU 核心和内存集成到一台服务器中，他们的每 CPU 核心成本和每 GB 内存成本都会降低。因此，通过购买更大的计算机，他们可以为购买较小虚拟服务器的用户提供更低的每 CPU 核心成本。

然而，第三个优势是最重要的——可管理性。为了支持虚拟化，每台机器的一小部分被专门用于虚拟机监视器——这是一个小型操作系统，用于管理服务器上的其他虚拟机。由于这些机器是虚拟机，而非真实的硬件设备，它们非常易于管理。操作员可以向虚拟机监视器发送命令，它可以在几分钟内立即设置一台新的虚拟机，克隆一个新的启动磁盘，并启动新的虚拟服务器。

过去，如果我想要一台新服务器，我必须先购买服务器，然后，在物理上位于控制台前，安装相关的操作系统和系统软件，最后将服务器搬到机架旁，插上电源并开机。我必须确保网络已连接并配置好。我可能还需要配置 BIOS 以支持无键盘操作。如果机器出现问题，我必须备份机器，找到一台新的物理机器，将备份复制到新机器上，安装新机器，并用新机器物理替换旧机器。

有了虚拟化机器，我就可以让虚拟机监视器来处理所有这些事情。我只需要有足够的额外服务器运行虚拟机监视器，这样当我需要一台新机器时，只需告诉虚拟机监视器为我创建一个新的虚拟机，并指定要使用的磁盘副本所在位置即可。

更棒的是，在大多数 IaaS 平台上，你甚至无需担心虚拟机监视器和容量问题。IaaS 供应商会为你处理这一切。IaaS 供应商提供了一个点击式界面，你只需登录网络管理控制台，就能启动虚拟服务器。你告诉它你想要多大的机器（即 CPU 核心数和内存大小），它就会为你分配一台服务器。你告诉它你想要在上面安装什么（要么是基础操作系统，要么是现有机器的克隆），它就会将其复制到启动磁盘上，并为你启动它。瞧！你就得到了一台可以立即使用的新服务器。

此外，对于大多数 IaaS 服务，如果出现物理硬件问题，服务商会为你处理。如果检测到严重的硬件问题，他们就会简单地关闭你的服务器，将其迁移到新服务器上，然后重新启动，并发送一封电子邮件通知你发生了什么。如果是较轻微的问题，一些供应商会向你发送通知，要求你在最方便的时候点击按钮执行迁移。

更棒的是其定价模式。大多数 IaaS 供应商都提供*按小时*计费的选项。你是否需要一台机器，但只用几个小时？使用 Linode，你可以获得一台 32 核、64 GB 内存的机器，每小时花费远低于 1 美元！

在第 3 章中，我们将了解设置云服务器的具体步骤。

## 2.2 平台即服务

另一种扩展应用的选择被称为*平台即服务*，缩写为 PaaS。使用 PaaS，PaaS *定义*一个平台供你在其上运行应用，而不是直接提供裸机让你随意运行任何东西。PaaS 供应商管理整个平台，你只需关注你的应用。

例如，Heroku 是一个受欢迎的 PaaS 供应商。Heroku 处理所有的服务器管理和维护工作。你甚至无法登录他们的机器！你只需将代码推送给他们，他们就会为你将其部署到他们的机器上。其他常见的 PaaS 供应商包括 Google App Engine、Windows Azure、Amazon Elastic Beanstalk 和 OpenShift。

使用 PaaS，你选择使用多少个“工作单元”，系统会将工作分配到所需数量的机器上（工作单元只是一个活跃的进程，每个 PaaS 供应商对其有自己的术语）。因此，PaaS 供应商负责*平台*（硬件、操作系统、已安装的应用），而你只需管理应用代码。

这理论上听起来很棒——你完全无需再管理服务器！所有这一切都在幕后自动神奇地为你完成。然而，现实是 PaaS 平台并不像看起来那么透明，而且 PaaS 供应商倾向于对其增值服务收取相当高的费用。



## 2.3 Docker

在当今备受瞩目的技术领域中，`Docker` 是一项新兴技术。`Docker` 应用介于 `IaaS` 和 `PaaS` 之间。本质上，一个 `Docker` 镜像包含运行应用程序所需的所有程序的完整安装。

它们可以像 `PaaS` 一样被管理、部署和扩展：你只需告诉服务你需要运行多少个实例，它就会负责将你的镜像部署到正确的位置；但与此同时，你拥有比 `PaaS` 大得多的控制权，这一点又类似于 `IaaS`。

`Docker` 的一个有趣特性是能够将不同的服务“链接”在一起。本质上，你为 `Docker` 容器提供或需要的不同类型的服务命名，然后可以使用这些名称告诉你的服务如何找到彼此。不过，这通常只是在应用程序初始设置阶段才会遇到的问题，之后，无论你是手动链接应用组件，还是使用某种高级工具进行链接，区别都不大。

`Docker` 本身是一项技术，而非供应商。有许多供应商允许你部署 `Docker` 应用，包括负责 `Docker` 技术的公司 `Docker Inc.`。

## 2.4 为什么选择 IaaS

本书将重点放在 `IaaS` 云模型上，原因如下。首先，`IaaS` 非常灵活。无论你想要运行何种类型的工作负载，也无论你想要在什么平台上运行，`IaaS` 都只是为你提供裸服务器。如何使用它们完全由你决定。这也意味着你不会被锁定在特定供应商的系统中。

其次，`IaaS` 的运行方式相当可预测。虽然存在差异，但大多数 `IaaS` 供应商的运营模式都非常相似：你选择一种机器规格，指定你需要的配置，然后按下按钮即可。这些服务的质量、成本和灵活性可能存在很大差异，但其基本操作模式都相当类似。

对于 `PaaS` 来说，你的选择则受限得多。首先，你只能使用供应商提供的平台。如果你使用 `Ruby on Rails` 编程，那么你只能使用 `Ruby on Rails PaaS`。虽然大多数 `PaaS` 供应商已经开放了他们的系统以支持数量众多的不同应用服务器，但限制依然存在。其次，你必须按照他们的平台配置方式编写代码。这通常意味着很少或没有本地文件存储、只能连接特定类型的数据库、只允许某些特定的平台选项或扩展，并且对服务器的配置方式几乎没有灵活性。有时候这没问题，但有时候这又过于束缚。

此外，`PaaS` 通常过于不透明。访问服务器日志通常很困难，而调试特定于服务器的故障几乎不可能。有时 `PaaS` 供应商会提供一些工具来帮助，但它们完全无法与直接在出问题的机器上进行调试相比。这种不透明性有时会导致灾难性的后果。例如，如果你购买了一个 `PaaS` 数据库系统，而不幸你的数据库损坏了，你的选择将极其有限。你基本上只能打电话给公司求助。有些公司在这方面响应非常迅速，但这仍然让我感到非常不安。

同时请记住，为了让 `PaaS` 系统正常工作，他们必须不断升级他们的系统。事实上，这正是你付钱请他们做的事情。然而，无法保证他们明天所做的升级不会意外地破坏你的应用。也许这个升级是必要的，但这不是你能决定的。

类似地，使用 `PaaS` 供应商意味着你必须按照*他们的*时间表进行升级。如果他们决定弃用其基础设施的某一部分，你就必须重写代码来适应。如果他们决定你使用的某个软件版本过时了，你就必须重写代码以使用新版本。简而言之，`PaaS` 意味着你失去了对技术的控制权，而 `IaaS` 则意味着你拥有完全的控制权。虽然使用 `PaaS` 可以省去某些麻烦，但这些节省通常会被其引入的新麻烦所抵消。

最后，`PaaS` 通常非常昂贵。例如，在 `Heroku`（一个常见的 `Ruby on Rails PaaS` 供应商）上，四个 CPU 核心每月花费 `$100` 美元。而在 `Linode` 上，一台四核机器的成本仅为每月 `$40` 美元，并且 `Linode` 的机器速度更快。我发现，大多数 `PaaS` 供应商的收费是同等性能的优秀 `IaaS` 供应商的三倍左右。`PaaS` 供应商为你做得更多，但这只有在你的开发团队没有任何系统管理经验时才值得，而且你通常会为了跟上 `PaaS` 平台的更新而付出同样的代价。

如果你的组织具备设置和维护服务器的技能（本书提供了一个学习如何做到这一点的良好开端），那么 `IaaS` 当前是利用云技术最简便、最灵活、最快且成本最低的方式，并且它不会将你锁定在特定供应商上。如果你真的决定想使用类似 `PaaS` 的系统，只需知道有许多开源的 `PaaS` 系统可以运行在你的 `IaaS` 服务器之上。



## 2.5 选择 IaaS 供应商

选择 IaaS 服务或供应商时需要考虑诸多因素。其中最重要的因素是服务的可靠性。如果服务宕机，再出色的 Web 应用也无济于事。如果无法访问数据，长期来看公司也无法盈利。因此，服务的可靠性应成为决策的关键考量。

这一点同样延伸至供应商处理工单的能力。任何服务都难免会遇到问题。如果一家公司无法及时响应或解决工单，那么你就不应将生产系统托付于它。

下一个主要因素是性价比。早期许多云基础设施公司过度强调云计算的灵活性，导致几乎所有云解决方案的价格都高得离谱。亚马逊网络服务（AWS）的云计算服务 EC2 就是一个典型例子。如前所述，一台 Linode 4 核 8GB RAM 的机器每月仅需 40 美元。而 EC2 上配置相似的机器（`c5.xlarge`）每月则需 122 美元。从历史上看，即使规格相同，EC2 的速度也往往远慢于 Linode。在云计算早期，EC2 是该领域屈指可数的大玩家，为了获得灵活性，你不得不支付高昂代价。就我个人而言，面对 EC2 的要价，我宁愿选择传统托管方案，直接租用物理服务器。对我来说，AWS 要求的灵活性根本不值这个价钱。

天平的另一端，则是一些价格低到令人难以置信的云服务。也就是说，从价格就能看出，他们不可能以这个价位提供可靠的长期服务。要么公司会倒闭，要么公司会涨价，要么服务最终会因过度订阅而降级至无法使用。CloudAtCost（`www.cloudatcost.com`）就是一个典型例子。使用 CloudAtCost，你只需一次性付费就能*永久*保留服务器。例如，花 70 美元就能获得 2 个 CPU 核心和 1GB 内存。但这*不是*月费，而是*一次性*费用！正如我所说，这价格好得令人难以置信。他们确实会收取每年 9 美元的账户维护费，但无论你有多少台服务器，这个费用都保持不变（设立此费用很可能是为了关闭那些不再维护的服务器）。

这类服务可能适合某些场景。例如，如果你想要一台开发服务器，那么偶尔网络中断或技术人员响应不及时都无关紧要。而且，如果哪天他们关门大吉，你也不会有太大损失。但我绝不会把业务押注在这种服务上。如果你想了解更多价格低得离谱的服务，可以访问`www.lowendbox.com`。

最后一个因素是灵活性。能够点击鼠标就设置好一台新机器是一回事，但如果你不能存储磁盘镜像，每次启动机器都必须重建，那该怎么办？这将是一个繁琐的过程，你会失去云计算的主要优势之一。而 AWS 在这方面表现出色。AWS 不仅提供云计算服务（EC2），还几乎为云基础设施的每一个方面都提供了配置、控制和自动化功能。AWS 提供可扩展的存储解决方案、视频转码服务、可扩展数据库、消息队列服务和搜索服务。这为你的集群提供了一个外部监控服务，能够随着负载增加自动生成新服务器来处理负载，并在网络负载降低时从集群中移除服务器并关闭它们。换句话说，AWS 带来的额外服务几乎是无限的。

然而，归根结底，AWS 的灵活性被 EC2 糟糕的性价比所掩盖。但值得庆幸的是，即使你选择其他供应商作为主要的 IaaS 提供商，也可以单独使用 AWS 的优质服务（例如 S3 和 CloudFront）。我们将在后续章节讨论如何实现这一点。

如果你还没猜到，我偏好的云供应商是 Linode（`www.linode.com`）。它的性价比无与伦比。原因有三。第一，设备更新。第二，他们只使用 SSD 驱动器（即固态硬盘——没有旋转磁盘），速度通常比普通磁盘快一个数量级。这可能是 Linode 性能最重要的方面之一。第三，他们实施了防止“吵闹邻居”的控制措施。

“吵闹邻居”是指与你共享同一台物理服务器的虚拟机，但它占用了计算机所有的 I/O 资源。在 IaaS 平台上，你无法选择（甚至不知道）还有谁在共享同一物理硬件。因此，使用能够防止资源霸占的服务非常重要。Linode 实施了一系列控制措施，防止任何单个虚拟机过度使用资源。这不仅有利于你自己虚拟服务器的性能，也意味着你无需担心自己会成为他人的“坏邻居”！

虽然 Linode 没有 AWS 那样的灵活性，但这种灵活性几乎无足轻重。在 Linode 上，大多数你在 AWS 上想做的关键事情都极其简单，而 AWS 的附加功能反而使其学习曲线陡峭且令人困惑。当有人为了获得 AWS 的灵活性而花费额外成本时（例如，创建一个系统以自动响应负载启动新机器），他们本可以用同样的成本在 Linode 上为集群增加足够容量，从而根本无需担心负载问题。

还有其他与 Linode 类似的服务——DigitalOcean 通常被认为是在同等价位下可与之媲美的服务。然而，我个人对 Linode 的体验足够好，以至于认为无需逐一研究所有其他服务。Linode 提供了我需要的一切，拥有易用的界面，并且价格实惠。因此，本书将专注于在 Linode 上开发云应用。

在选择供应商时，仔细阅读其服务条款以确保你的预期用途与之兼容也至关重要。不仅某些服务禁止特定的工作负载（例如，一些服务禁止大规模电子邮件营销），许多服务还对你扩展网络的速度有限制。这主要是为了防止滥用，但提前了解这一点非常重要。无论如何，在选定供应商之前，你可能需要联系潜在的云供应商，确保你的使用计划符合他们的服务准则。


## 2.6 一些重要的术语

在深入探讨之前，我想先澄清本书中使用的一些术语。使用 IaaS 时，你租用的大致是机器。它们是*虚拟*机器（即真实机器划分出的分区），但抽象来说，它们本身仍可被视为机器。在云计算术语中，它们通常被称为*节点*。由于它们为网络上的其他机器和/或用户提供服务，因此也被视为*服务器*。因此，节点常被称作 VPS——虚拟专用服务器。在本书中，`machine`、`node` 和 `server` 这三个术语基本可以互换使用，不过选择哪个特定术语通常取决于你当时的用法。

*集群* 是一组协同工作的节点。因此，我们正在云端构建一个集群。一种为在云端实现可扩展性而构建的服务被称为*云集群*，或者简称为*云*。

*可扩展性* 是云计算中的一个重要术语。可扩展性与性能不同。性能指的要么是效率，要么是原始速度。可扩展性指的是系统快速扩展其处理能力的能力。例如，一个每秒能处理 2,000 个请求的程序可能被视为高性能，但如果这个程序的速度无法通过增加节点来提升，那么它就不具备可扩展性。

另一方面，即使是低性能的程序也可能是可扩展的。如果我的程序每秒只处理 1 个请求，但我每增加一个节点，它能处理的请求数量都会增加，那么我的程序就是可扩展的。如果我将这个低性能程序扩展到 4,000 个节点，它将能够处理比之前那个最多每秒处理 2,000 个请求的程序更多的请求。整个集群（拥有 4,000 个节点）现在可能会因为整体吞吐量而被视为高性能，即使其各个部分并非高性能。

通常，可扩展性取决于你*并行化*任务的能力。并行化指的是任务能够彼此独立运行，甚至在其他机器上运行的能力。两个任务需要相互协调的程度越高，它们的可并行化程度就越低。在设计面向可扩展性的应用程序时，你需要特别关注系统不同部分需要协调的方式，并力求最小化或消除这些协调点的影响。

希望你所构建的东西*既*是高性能的（即使节点数量较少）*又*是可扩展的（以便增加更多节点能提升性能）。让应用表现良好，需要寻找低效代码并用快速代码替换，或者重写应用以避免使用低效代码。让应用具备可扩展性，则需要寻找阻碍并行化的*瓶颈*，并将这些瓶颈替换为能够扩展到多个节点的代码，或者重新设计你的架构，使瓶颈根本不会出现。

本书的重点是可扩展性，不过也会兼顾性能。

### 告别服务器机房

这里有一个关于我不再怀念处理服务器机房事宜的有趣故事。在 20 世纪 90 年代末，我曾在一家托管自己服务器的公司担任程序员/系统管理员。这些服务器不在标准的服务器机架里，而是放在金属线架上。主要的面向外部的服务器就在顶层架子上。

这台服务器需要增加一些硬盘空间，于是我给它安装了一个外部硬盘组。那时，连接外部存储到服务器的主要方式叫做 SCSI（发音为“scuzzy”），并且问题多多。过了一段时间，服务器访问硬盘组开始出现大量问题，于是我去排查。

由于服务器在顶层架子上，我站在凳子上操作机器。我花了大约半个小时试图找出问题所在。要知道，这发生在我们的主外部服务器上，所以每一分钟都很宝贵，因为我们的（非常活跃的）网站正出现问题。

于是，我设法把所有数据从新的硬盘组中移出，然后断开硬盘连接，准备把它们拿回办公室看看问题出在哪里。然而，我站在凳子上摆弄电脑太久了，以至于忘了自己还站在凳子上。所以，我抓起硬盘组就想走出房间，至少我是这么想的。庆幸的是，我和硬盘组都没有摔伤，但我的自尊心确实受到了伤害。

虽然回忆这样的故事很有趣，但我很高兴那些日子已经过去了。
