# 第一部分：解耦式 Drupal 基础

第 1 部分

在第一部分中，我们将为解耦式 Drupal 建立重要的概念基础。这种范式虽广受欢迎，但也并非没有风险。首先，我们将审视不断变化的网络如何通过为其他技术中的数字体验敞开大门，影响了内容管理系统的演变。服务端和客户端（尤其是通用 JavaScript）的同步变革，甚至鼓励网络从业者探索解耦式 CMS 架构，而不仅仅将其用于原生应用。

我们将解耦式 Drupal 定义为将 Drupal 用作供其他应用程序消费的内容服务，并在此过程中识别出两种主要的架构范式。第一种是完全解耦式 Drupal，它将 Drupal 的默认前端与消费者应用程序完全分离。这种方式不仅越来越多地用于满足原生应用和物联网应用的需求，也用于 JavaScript 应用。另一种是渐进式解耦 Drupal，它通过将 JavaScript 框架插值到 Twig 驱动的前端中，提供了一种折中方案。

最后，在深入探讨解耦 Drupal 的用例和动机之后，我们将分析这些方法的一些关键优缺点。这些考虑因素不仅涉及开发者，还包括那些不可避免地会塑造和操纵最终体验的内容编辑者、站点构建者、站点管理员以及最终用户。由于解耦式 Drupal 相对不成熟，我鼓励您在项目探索阶段仔细评估这些风险与回报，因为这些架构决策可能会在后续过程中产生不成比例的影响。
